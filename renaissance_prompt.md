# 技术觉醒深度学习框架设计方案

# 【一、前言】

我们的团队正在开发一款轻量级、高性能、多后端、可重构、跨平台的自研深度学习框架，名为**技术觉醒**（tech-renaissance）。

首先，这个框架并不是我们的第一个框架，也不是第一个版本。几年前我们开发过轻量级的深度学习框架“湍流”，具备最基本的深度学习框架功能，但只支持CPU，不支持其他后端，且优化较差。今年开始开发的技术觉醒框架（最新版本是到V2.4.6），此前已取得阶段性成果，实现了基本模块和核心算子，FP32的性能略优于PyTorch。因此我们已经具备一定的经验。

但是，技术觉醒框架的第2代的基本设计存在一些问题。主要在于此前的考虑是将其作为一个小型的试验项目，对可扩展性考虑不足，也没有大胆设想跟主流的开源深度学习框架进行性能上的竞争，整体设计偏保守和狭隘。主要局限性体现在：第一，只规划了FP32和INT8、INT32，没有考虑AMP的实现。第二，只考虑了单卡训练，没有考虑多卡的数据并行的实现。第三，张量的数据布局采用传统的NCHW，而当前GPU对NHWC的加速支持更为强大。第四，对于多后端尤其是边缘设备推理的考虑不够充分。第五，第2代框架设计初衷直接定为开源，没有充分考虑闭源的需求。这五点局限性促使我们重新设计技术觉醒框架，而非在原来的基础上进行修改，因为这涉及到数据类型、数据布局、算子库的调整，改动极大。对于AMP、数据并行、NHWC的支持，都是出于训练性能考虑，毕竟在大规模训练中，主流深度学习框架一般会把这些选项都开启，以获得最优性能。如果我们要在性能上跟主流深度学习框架比拼，这几种功能是必须支持的，否则性能上的优势就没有说服力。此外，第2代的技术觉醒框架还存在一个“后端”与“器件”术语混淆的小问题，显得不够专业。

我们现在要设计开发的，是**技术觉醒3**。

需要特别强调的是：第一，我们拥有一定的经验和基础，实现一个性能良好的轻量级自研深度学习框架、甚至在部分指标上比肩主流框架并非痴人说梦。第二，这依然只是一个轻量级深度学习框架的初期版本，它并非产品，它当前仅用于演示、教育、交流、学习、研究，它并不追求功能的面面俱到，只求在专注的领域上实现足以演示的功能和较好的性能。

简单来说，我们的自研深度学习框架当前**只做CV，而且只做CNN，而且只实现最核心最常用的那一部分模块**，但是，**它最好要在ResNet-50的训练和推理上比肩和超越PyTorch**。

因为有了此前的开发经验，我们对功能定义、工具选型有了一个相对而言更为完善而且实际的方案。为了节约时间、人力、物力等成本，我们依然会严格控制深度学习框架所需要拥有的功能，避免功能无序扩张、贪大求全而导致多而不精，甚至项目烂尾。

尽管有了前面的经验和强大的第三方库，以及有AI编程的助力，但开发工作量依然非常巨大。下面介绍我们的基本设计思路。



# 【二、基本设计】

## （一）基本属性

1. 这是一个静态图深度学习框架。
2. 本框架当前**只做CV，而且只做CNN，而且只实现最核心最常用的那一部分模块**。
3. 本框架的开发者包括人和AI，整个过程会充分利用AI编程，也会借助AI进行设计和决策。
4. 本框架**以开发者为中心**。
5. 本框架需要适配的操作系统是Windows（主要是Windows11）和Linux（主要是Ubuntu 24.04 LTS，当然，还需要支持边缘设备的Linux系统）。
6. 本框架会用C++开发底层，在功能实现并完成验证后，用PyBind11进行封装，最后提供C++和Python两套API。
7. 本框架不能依赖于其他的任何深度学习框架。但是，它可以使用包括高性能计算库在内的多种第三方库。第三方库尽量使用vcpkg来管理。
8. 由于会用到各种第三方库，本框架采用Python脚本来自动配置依赖。
9. 本框架支持的Tensor的数据类型包括：FP32、BF16、INT32、INT8，总共4种。
10. 本框架采用的数据排列格式是**NHWC**。CPU和GPU上都是。目前越来越多的框架都在强调对 **NHWC (channels last)** 的支持。根据 NVIDIA 的白皮书，ResNet-50 在 A100 上使用 NHWC + TensorCore 可获得 1.3-1.8× 的端到端加速。此外，关于“最符合人类思维的维度顺序”，我问了10个AI，10个AI都回答说是**NHWC**。原生支持NHWC，我们将同时拥有性能和易用性的优势。
11. 本框架的六大板块是：数据（data）、模型（model）、训练器（trainer）、器件（device）、任务（task）、工具（utils）。
12. 本框架所追求的“可重构”主要体现为算子、器件支持的模块化、插件化设计，后期将可以按需选择部件，将框架裁剪成一个更小的子集。举例来说，可以只支持全连接而不支持卷积，可以只支持CPU而不支持GPU，用不上的模块可以不勾选，使得框架更为精简，且去掉多余模块后，框架的其他功能依然不受影响。
13. 本框架的最核心特色之一是**后端解耦**，目的是方便后端开发者进行扩展，尤其方便自研人工智能芯片的开发者。后端解耦的主要表现包括：
    1. Tensor本身不定义任何运算功能，所有运算由其所在的器件执行，需要显式调用器件对象的方法。
    2. 各后端分别独立实现算子。
    3. 每个后端并不需要实现所有的算子，当后端不支持该算子时，会运行时报错。
14. 本框架计划支持的器件包括：**x86 CPU**（必须支持Intel和AMD，但测试主要是在Intel Core i9-14900HX和Intel Xeon Gold 6530）、**NVIDIA GPU**（主要测试A10、RTX 4060、RTX5090）、**摩尔线程GPU**（主要测试MTT S80）、**FPGA**（主要测试Intel SoC）、**ARM**（主要测试树莓派5）。除此以外，本框架还将支持模型导出，导出的模型将会在Intel NCS2、华为Atlas 200i DK A2上进行INT8推理试验。这里我们非常明确，Intel NCS2、华为Atlas 200i DK A2属于专用硬件，没有必要去替厂商开发专用硬件的支持，而是应该直接使用它们的工具进行推理。
15. 本框架将支持模型的8-bit量化，包括PTQ和QAT。量化后的INT8模型将可以运行在x86 CPU、NVIDIA GPU、FPGA、ARM。
16. 自动混合精度（AMP）训练主要使用BF16和FP32。
17. 支持的数据集只需要包括：MNIST、CIFAR-10、ImageNet。
18. 优化框架性能的最关键思路是**避免内存的反复分配**以及**零拷贝**。我们在技术觉醒2中设计了**into型方法**，也就是将运算的结果写入指定的张量而非返回一个新的结果张量（比如void mul_into(const Tensor& a, const Tensor& b, Tensor& result)就是把张量a、b逐元素相乘的结果写入同形张量result），这种方法在实践中取得了接近50%的提速，非常成功，是我们的技术觉醒2框架在运算性能上超越PyTorch的关键。
19. 考虑到时间人力成本，此前的技术觉醒2的规划是“暂不实现内存池”。但在技术觉醒3中，内存池的实现值得重点考虑。
20. 本框架的C++命名空间是tr。Python的包名是renaissance，导入后的默认简称也叫tr。
21. 本框架的主头文件是renaissance.h。测试样例一般只需包含主头文件和必要的标准库头文件，而不包含框架内部的其他头文件。
22. 由于Windows下CUDA的兼容性问题，我们在Windows下使用MSVC作为默认编译器。对于Windows下的与CUDA无关的功能，可使用MSYS2的g++进行编译。而在Linux，默认使用g++。
23. C++采用CMake来构建项目。
24. 出于未来的利益考虑，我们决定在开源问题上暂时**有所保留**。技术觉醒3框架将分为**完整版**和**开源版**。完整版将托管在Gitee的闭源仓库，而开源版将在完整版功能开发验证完成后制作，进行合理裁剪，阉割部分功能，然后公开发布到GitHub和Gitee。
25. 本框架的完整版将在编译成库之后进行Docker分发，通过Docker提供试用。
26. 本框架已有相应的官方域名（技术觉醒.中国、tech-renaissance.cn），且已备案完成，在开发基本完成后，将上线官网，并在官网提供介绍和使用教程等。
27. 本框架的核心演示目标是ResNet-50和MobileNetV2，原则上只要是这两个模型没有的运算，都暂时不需要支持。
28. 本框架需要借助PyTorch来调试，所以至少在早期是需要与PyTorch的功能对齐的，也就是，同样一个操作，我们要得出跟PyTorch一致的结果。
29. 代码风格尽量遵循业界规范（如Google C++ Style Guide），但我要求全部注释使用中文，全部打印输出及异常信息都使用英文，代码里不能包含任何emoji，以免碰到兼容性问题。
30. 图像预处理考虑采用ermig1979的Simd库来实现，运算全部在CPU上进行，要充分利用多线程。
31. 本框架主打易用性，API要简洁易用，并且允许使用YAML进行快速配置，像NVIDIA TAO一样实现低代码的模型训练。
32. “三驾马车”：与本框架一道开发的项目还包括AI编译器**“算力场**”（Compute Power Field，CPF）、基于FPGA的AI芯片“**FI-0420**”，它们主要是用来演示我们的框架对AI芯片研发的助力。但重心依然是在深度学习框架上。
33. 支持模型导出、数据导出（张量导出）和数据集导出。模型导出，我们可以有自己的格式（.mdl），然后可以导出为ONNX（这个非常重要！必须实现）。数据导出，我们有自己的格式（.tsr），然后还要支持导出为.pth（需要安装pytorch，调用脚本转化）、.npy（需要安装numpy，调用脚本转化）。数据集导出，其实就是保存数据集里面所有的样本张量和标签，这是为了统一格式，给用户更好的体验，我们用自己的格式（.dts）。
34. 支持保存checkpoint（.ckpt），支持断点续训，也就是能完全保存训练状态，包括模型结构、模型参数、模型超参数、训练器内保存的动量、训练执行到哪个批次等。
35. 深度学习模型中所有的“层”都属于模块类（Module）的子类，根据我分析的需求，我们只需要实现以下几个层：卷积层（Conv）、全连接层（Linear）、ReLU激活函数（ReLU）、BN1D、BN2D、最大池化（MaxPool）、全局平均池化（GAP）、Flatten。
36. 损失函数只支持交叉熵。
37. 关于ResNet结构里最重要的Shortcut，我的观点是，通过“并行容器”来实现。容器就是决定各个层的连接的结构，它分为“顺序容器”（Sequential）和“并行容器”（Parallel），容器可以装各种不同模块，顺序容器里面各个层都是首位顺序连接，而并行容器里面各个层都是享有相同的输入，然后最后把输出加起来。
38. 需要支持模型结构打印，也就是在控制台输出文本，表明这个模型的结构是什么。
39. 需要支持标签平滑（label smoothing）。
40. 支持三种优化器：SGD with momentum、Adam、AdamW。
41. 支持七种学习率调度器：常数学习率、StepLR、MultiStepLR、指数、余弦退火、带热启动的余弦退火、用户自定义函数的学习率调度器。
42. 支持算子融合。如果卷积层是C、BN层是B、全连接层是L、ReLU层是R，那么咱们的框架支持的算子融合就是4种：CB、CBR、LR、LBR。
43. NVIDIA GPU上一定要使用CUDA，C++开发。而不要使用PyCUDA或者OpenCL之类。
44. GPU的显存，RTX 4060是8GB，摩尔线程MTT S80是16GB，A10是24GB，RTX 5090是32GB。我们用于研发和测试的主要平台是配备两个RTX 5090的服务器。
45. 完成基本开发后，我会借助AI来生成文档和示例程序，确保用户在拿到我的文档和示例程序后，可以通过对AI下达命令，就让AI自动生成能运行于技术觉醒框架的代码。
46. 关于workspace目录管理：我们在项目根目录下设置workspace目录，临时文件统一放在这里。
47. 演示所使用的ResNet相比官方原版有两个改动：一是最大值池化，我们采用标准的kernel=2、stride=2、padding=0，以简化硬件实现；二是最后的全连接层，我们去除bias。

## （二）简化与限制

为了加快开发速度、避免bug、减少后续硬件开发的难度，我需要对框架做一些限制和简化（尽管它们是开源深度学习框架不常有的）

1. 在技术觉醒框架中，所有的张量的维度上限都是4维。
2. 卷积层默认是2D卷积，所以Conv2d我们就直接叫Conv。同理，最大池化和全局平均池化也是默认2D。
3. 我需要减少“特殊情况”的数量，以免考虑得太复杂，所以，我强制要求所有的卷积核都是“方形的”，也就是，只允许n×n卷积，而不允许m×n卷积（m≠n）。这样一来，我们在给卷积层传递参数的时候，关于H和W就只需要传递一个参数即可，因为它们必须相等。
4. 由于后期我会在FPGA上实现，而FPGA不可能轻易实现任意大小的卷积核，所以我进一步限定卷积核的长宽只能是以下4种：1×1、3×3、5×5、7×7。
5. ResNet-50的卷积核的输出通道数最大是2048，那我们就假定允许的卷积核最大输出通道数不会超过2048，超过就报错。
6. 全局平均池化一律把特征图的长宽压缩到1×1。
7. 所有的卷积、池化的stride都只能是1或2。
8. 卷积层、全连接层默认禁用bias，因为已有研究表明bias对模型的能力影响极小，且BN层可以完全替代bias的功能。

## （三）后端与器件

为了明确说明我们的框架所需要支持的后端与器件，以及我们的框架导出的模型需要能够适配的后端和器件，我列了一个表：

| 序号 |            器件（Device）             |  后端（Backend）  |              备注              |
| :--: | :-----------------------------------: | :---------------: | :----------------------------: |
|  1   |                x86 CPU                |  Naive / oneDNN   |                                |
|  2   |                  ARM                  |  Naive / XNNPACK  |     仅需支持FP32和INT8推理     |
|  3   |              NVIDIA GPU               |       cuDNN       |                                |
|  4   |              摩尔线程GPU              |       muDNN       |     仅需支持FP32训练和推理     |
|  5   | FI-0420（**自研**的基于FPGA的AI芯片） | trDNN（**自研**） |        仅需支持INT8推理        |
|  6   |         华为昇腾Atlas 200i DK         |       CANN        | 仅用于验证和部署导出的INT8模型 |

需要注意的是，对于Atlas、NCS2，只是用于验证和部署导出的INT8模型，本身不需要算作是框架“支持的”硬件，框架不需要感知它们的存在，不需要运行在它们上面，也不需要从它们获取结果和反馈。

对于自研的基于FPGA的AI芯片，我的设想是打造一个“技术觉醒3（深度学习框架）→CPF编译器（AI编译器）→trDNN（FPGA/SoC上的自研计算存储模块的驱动及开发套件）→FI-0420（AI芯片）”的工具链。

## （四）演示的功能

1. **图像分类**：演示不同的神经网络对3种数据集的分类，具体包括：
   1. MNIST：MLP（3层）、LeNet-5、TorrentNet（类ResNet卷积神经网络，此前“湍流”框架的演示模型之一）
   2. CIFAR-10：NIN（Network-in-Network，论文准确率91.19%）、ResNet-110（论文准确率93.39%）
   3. ImageNet：ResNet-50、ResNet-18、MobileNetV2、VGG-16、AlexNet
2. **量化**：将模型量化为INT8，具体包括：
   1. PTQ，对ResNet-50进行量化
   2. QAT，对ResNet-50进行量化
3. **AMP**：使用BF16加速ResNet-50的训练
4. **数据并行**：在两个A10上训练ResNet-50
5. **迁移学习**：将ImageNet2012预训练的ResNet-50用于CIFAR-10识别
6. **超参数优化&集成学习**：演示如何逐步优化超参数以提升CNN对MNIST的识别准确率。最后是用5个TorrentNet组成ensemble对MNIST进行分类（在PyTorch上的实现达到99.91%的SOTA级识别准确率）
7. **断点续训**：保存checkpoint，实现ResNet-50断点续训
8. **多后端**：演示摩尔线程MTT S80上的ResNet-50训练和树莓派5上的ResNet-50推理
9. **模型导出**：演示将MobileNetV2导出为ONNX，然后分别编译运行在Intel NCS2和华为Atlas 200i DK上
10. **可重构**：演示对框架进行裁剪，选择部件，形成一个体量更小、更精简的框架
11. **自研AI芯片及工具链**：演示如何将ResNet-50通过自研的CPF编译器部署到自研的基于FPGA的AI芯片FI-0420上
12. **低代码**：演示TASK模式的极简API写法、演示用YAML配置ResNet-50的训练
13. **性能对比**：对比技术觉醒3与PyTorch在几种核心运算上的性能（FLOPS）、ResNet-50的全精度训练速度和AMP训练速度、在CPU/GPU/树莓派上的推理速度

# 【六、开发原则】

## 一、总体原则

1. **以开发者为中心**：优先保证开发效率和可行性，不盲目追求用户友好性；功能不无序扩张。
2. **最小可行产品（MVP）先行**：限制功能，先把最小可行产品实现好，功能如前文所述，非必要不添加。
3. **结果对齐验证**：开发过程中，关键运算的结果要尽量与 PyTorch 输出结果在数值上高度一致（误差 ≤ 1e-5），建立自动化测试体系。
4. **静态图 + 手动求导**：在静态图模式下，整个计算图的内存需求可以在执行前被**完整分析**。不实现自动微分引擎；所有层需手动实现前向与反向传播逻辑。

## 二、代码与工程规范

1. **命名与风格**：
   - C++ 代码遵循 **Google C++ Style Guide**。
   - 所有注释使用 **中文**，所有打印输出（如日志、错误信息）使用 **英文**。而且，**注释和输出当中都不能包含emoji**。
   - 主头文件为 `renaissance.h`，命名空间为 `tr`；Python 包名为 `renaissance`，常用别名为 `tr`。
   - 头文件使用`#pragma once`防止重复包含。
   - 括号：左括号不换行，右括号独占一行。
   - 注释：类/函数——**使用Doxygen格式**（如`/** ... */`），包含功能描述、参数说明；复杂逻辑——行内注释解释（中文）。
   - 版本号：V+三个整数。比如：V3.1.4。不使用多余的前置0。
2. **文件组织**：
   - 参考材料中给出的目录结构（`include/renaissance/`、`src/`、`tests/` 等）。
   - 头文件与源文件按功能要素（数据、器件、模型、训练器、任务、工具）分目录组织，结构镜像。
   - 测试样例产生的临时文件统一放在根目录下的workspace文件夹中，且通过WORKSPACE_PATH宏来引用。该文件夹会自动创建。
   - 所有测试样例产生的临时文件，除特别声明以外，都应该在测试结束时删除
3. **构建系统**：
   - 使用 **CMake** 作为唯一构建工具。
   - 通过 `option()` 实现“可重构”编译（如 `TR_BUILD_CONV_MODULE`）。
   - 分层CMake结构：根`CMakeLists.txt`管理子目录，子目录（如`src/data/CMakeLists.txt`）负责局部构建。
4. **依赖管理**：
   - 使用 `vcpkg` + `find_package`。
   - 核心依赖：CUDA 12.x、cuDNN 8.x、cuBLAS、oneDNN、XNNPACK、Simd、libcurl。如无特殊情况，不要添加其他第三方库！
   - 特别强调：不可以使用类似<Windows.h>这种平台相关的头文件！

## 三、核心架构规则

1. **数据类型**：
   - 张量里的数据类型DType只支持4种：FP32、BF16、INT32、INT8
2. **张量与内存模型**：
   - 所有张量的维度上限为 **4 维**
   - 采用 **句柄式内存管理**：`Tensor` → `Storage` → `Device`。
3. **器件抽象**：
   - 定义 `Device` 基类，派生 `CpuDevice`、`CudaDevice` 等。
   - **尽量调用 cuDNN/cuBLAS**，不要手写 CUDA Kernel。
   - 器件通过 `DeivceManager`（单例）全局访问。
4. **模块与算子**：
   - 所有层继承自 `Module`，不单独设立“算子类”。
   - 支持的层有限制，详见上文。
   - 卷积核限制为 **方形**，且仅支持 **1×1、3×3、5×5、7×7**，输出通道 ≤ 2048。
5. **算子融合**：
   - 支持 4 种融合：CB、CBR、LR、LBR。
6. **任务（Task）机制**：
   - 支持原始的Forward-Backward写法，也支持通过Task机制简化训练推理工作流。
7. **量化支持**：
   - 仅支持 **INT8**，采用 **对称量化 + 逐通道**。
   - 实现 PTQ 与 QAT。

## 四、可重构与可扩展性

   - 通过 CMake 选项控制模块是否编译（如 `TR_BUILD_CONV_MODULE`）。

**问：你提到框架支持ONNX的导出。请问反向导入（ONNX→tech_renaissance）是否也在规划内？**

答：暂不包含从ONNX导入模型。只能从自己的.mdl文件导入。

## 问题2

**问：关于C++与PyTorch的交互，这涉及到了数据从Python（PyTorch）到C++（技术觉醒框架）的传递。为了最小化IO开销，我们是否计划使用如“共享内存”（Shared Memory）之类的IPC（进程间通信）机制来传递预处理好的张量数据，还是采用其他方式（例如临时文件或网络传输）？**

答：我的设想是，采用临时文件。我们的框架并不依赖PyTorch，只是在开发阶段根据需要借助PyTorch进行运算的验证，这种情况下对速度是没有要求的。通过临时文件来交互，实现简单、依赖极少，并且在发布时可以轻松把这个功能删除。

## 问题3

**问：为什么技术觉醒2框架的“后端”在技术觉醒3框架要改为“器件”？这两者有什么不同？**

答：技术觉醒2框架有CPU后端类和CUDA后端类。但是这两者其实都应该算是器件。我们在早期设计中命名出现了偏差。事实上“器件”指的是具体的硬件，是进行张量运算和存储的地方，而“后端”描述的是“用什么执行引擎 / 实现体系去跑算子”。用ChatGPT的描述来说，器件是执行地点，后端是执行方式。用Gemini的比喻来说，器件是赛车，后端是赛车手。我们前文已经把器件和后端的对应关系厘清了。事实上，多数情况下我们的框架都为一个器件选定一个最合适的后端实现，只有在使用CPU时，一个器件对应多个不同后端，也就是有SIMD实现的后端和朴素实现的后端。我们采纳Gemini的建议，对于CPU器件要**动态选择 Backend**（检测到是 Intel CPU 就加载 OneDNN 后端，检测到是 ARM 就加载 XNNPACK 后端）。在技术觉醒3框架编程时，要非常注意，因为我们基本上只在设计时提后端的概念，而编程时讲的都是器件。

## 问题4

**问：技术觉醒3的开源版和完整版有什么不同？**

答：技术觉醒3的开源版是在完整版开发基本完成且验证通过后，从中删减高级功能而成的。我们分享基本的设计思路和底层的实现代码，证明我们的原创性和架构优越性，也激发广大开发者的参与度，扩大本框架的知名度。但是方便而强大的最新的高级功能的实现我们暂时保留。我打算把完整版叫做renAIssance-3F，开源版叫做renAIssance-3O。具体来说：

完整版：开源版的所有功能、BF16及AMP、CPF编译器与FI-0420加速器、摩尔线程GPU的FP32支持、Python接口、模型导出、可重构、YAML配置。

## 问题5

**问：你多次提到采用ResNet-50的“金标准”来演示和benchmark，你对所要实现的性能和精度有什么预期？**

答：这个问题我们此前回答过，说是对标PyTorch，而且对标的也只是我们实测的数据。现在随着开发的进行，我们的调研进一步深入，我们认为真正有意义的benchmark还是MLPerf。更具体来说，是MLPerf V1.0的图像分类任务单机冠军。选择V1.0而不是后续版本，原因是我们的硬件只有A100，没有H100等更高级的硬件，而A100取得最佳成绩就是在V1.0。在MLPerf V1.0的图像分类任务中，浪潮公司用8个A100，花了27.38min训练好了ResNet-50，Top-1验证准确率达到了MLPerf V1.0规定的75.9%。这就是我们要严格对标的对象。作为一个真正要以性能作为重要指标、并且立志超越PyTorch的深度学习框架，选用世界一流的公认标准是理所当然。我们的最终演示也会选择装有8个A100的云服务器平台，以此达到公平对比，更有说服力。截至2026年1月1日，我们已经确定好了进行演示的算法和超参数，我们将可以在30个epoch内完成训练。这就意味着我们需要达到54.76s/epoch的速度。另一方面，我们还跟FFCV对标，FFCV是在21.7min内完成24个epoch的训练，使Top-1准确率达到75%以上。我们的超参数配置已经可以在24个epoch内达到75%的Top-1准确率。后续我们将主要进行这两个演示——一个是达到75.9%（对标浪潮），一个是达到75.0%（对标FFCV）。事实上，浪潮采用的是混合图框架mxnet，而FFCV采用的是动态图框架PyTorch，我们挑战的不是单一的深度学习框架，而是业界顶尖的所有深度学习框架。

## 问题7

**问：关于 BF16 混合精度训练的主权重管理**

**背景：** 文档提到支持 BF16 和 FP32 的 AMP 训练（2.1.16/2.4.3），且 SGD/Adam 优化器需要更新参数。
**问题：** 在显存中，你们的策略是维护一份 FP32 的 Master Weights（主权重）用于更新，还是直接在 BF16 权重上进行更新？

*   如果是前者（类似 PyTorch 做法），`Optimizer` 是否已经接管了 FP32 Master Weights 的同步拷贝逻辑？
*   鉴于你们强调“内存分配的优化”，这种双倍权重的存储策略是否已经在显存预算（RTX 4060 8GB）的考量范围内？

答：维护一份FP32的主权重用于更新，每次更新后立即生成一份BF16副权重用于前向和反向传播。我们的思路是module拥有权重，而optimizer保存动量等信息，并接管全部权重更新运算。关于双倍权重的存储（准确来说是双份权重存储），我们已经用PyTorch做过实验，对于双卡训练的ResNet-50，在每个GPU的batch size为256的情况下，每个GPU上的显存占用大约是13620M，远小于A10（24GB）、RTX 5090（32GB）的显存上限。而我们并不打算使用个人笔记本电脑的GPU（RTX 4060）来进行ResNet-50的训练，一般都不会这么做。因此，**显存容量大小不是瓶颈**。这种双份权重存储的策略，一个重要的好处是**节省显存带宽**，因为我们不需要频繁地从显存读32位权重，而是读16位权重。

## 问题8

**问：您计划在两个A10上实现数据并行训练。这是一个非常核心且复杂的功能。请问您计划采用何种技术方案来实现梯度的同步？**

*   是计划自己从底层（如Socket编程）手动实现一个Parameter Server或Ring-AllReduce算法吗？
*   还是计划在上层封装一个成熟的通信库，例如NVIDIA的**NCCL**（NVIDIA Collective Communications Library）？

选择NCCL可以极大简化开发，并保证极高的通信效率，因为它针对NVIDIA GPU和NVLink/NVSwitch进行了深度优化。但它会增加一个外部依赖。如果选择自研，虽然自主可控，但开发和调试的成本会非常高，且在多节点扩展时可能会遇到性能瓶颈。您对此的权衡和决策将直接影响多卡训练的性能和开发进度。

答：经过慎重权衡，为了确保能够达成“训练速度不低于PyTorch”的核心指标，我们决定**采用 NCCL (NVIDIA Collective Communications Library) 作为NVIDIA GPU上的通信后端，并对其进行轻量级封装。**

我们的决策逻辑如下：

1. **性能决定论**：我们的目标是在RTX 5090/A10上通过PCIe进行高速互联（需要说明的是，包括A10在内的绝大多数国内云服务提供商的GPU，都不支持NVLINK）。NCCL是NVIDIA官方针对其拓扑结构（包括NVLink、PCIe Switch）做了汇编级优化的库，能够轻松跑满带宽。如果采用自研Socket甚至Ring-AllReduce算法，不仅开发周期长（可能长达数月），且难以达到NCCL的拓扑感知性能，这会直接导致我们在“性能对比”环节落败。
2. **符合“后端解耦”架构**：我们将定义一个抽象的 `Communicator`（通信器）类。
   - 在 **NVIDIA GPU** 上，`CudaCommunicator` 实现调用 NCCL 接口。
   - 在 **CPU** 上，未来可视情况接入 MPI 或 Gloo（但目前暂不实现）。
   - 这种设计没有破坏框架的独立性，而是将 NCCL 视为类似 cuDNN 的“核心加速组件”。
3. **依赖可控**：NCCL 虽然是外部库，但它通常随 CUDA 生态一起分发，对于使用 NVIDIA 显卡的用户来说，这是一个“隐形”的依赖，不会增加额外的配置负担。

**技术路线**： 我们将实现一个简单的 `AllReduce` 接口。在反向传播结束后，`Trainer` 触发 `Communicator::all_reduce(gradients)`，NCCL 在 GPU 显存内直接完成规约，完全绕过 CPU 内存，实现零拷贝同步。

## 问题9

**问：多卡数据并行的通信后端选择**

**背景**：文档提到"支持双 RTX 5090 的数据并行"，但未明确通信方式。

**疑问**：

1. 是否采用 **NCCL**（NVIDIA Collective Communications Library）作为 AllReduce 后端？
2. 如果自研通信层，是否基于 CUDA IPC（Inter-Process Communication）实现 zero-copy 的梯度同步？

**技术影响**：ResNet-50 在双卡训练时，通信开销占比约 15-25%，通信后端的选择直接影响 scaling efficiency。

答：我们的明确规划如下：

1. **关于 NCCL**： 是的，我们将 **100% 依赖 NCCL** 作为 NVIDIA GPU 的 AllReduce 后端。这是目前实现 ResNet-50 高效扩展（Scaling Efficiency > 90%）的唯一工业级标准路径。
2. **关于 CUDA IPC (Inter-Process Communication)**： 我们 **不需要** 手动编写基于 CUDA IPC 的代码。NCCL 内部已经封装了 CUDA IPC（用于单机多卡 P2P 通信）和 GPUDirect RDMA（用于多机通信）。手动实现 CUDA IPC 属于重复造轮子，且极易出现死锁或显存同步 bug，不符合我们“以开发者为中心，通过最小可行产品验证性能”的原则。

## 问题10

**问：关于“任务（Task）”系统的设计，目前看起来非常简洁高效。  **

> 是否打算在Task体系中引入**图优化Pass机制**（如算子融合、常量折叠、内存复用分析），形成类似“任务→图→执行计划（Execution Plan）”的中间抽象，以便未来AI编译器CPF复用？

答：**是的，这不仅是打算，而是实现高性能静态图框架的必经之路。**

虽然我们在代码层面看似是手动堆叠 `Layer`，但由于我们明确定义了 **静态图** 属性，我们将在 `Task.compile()` 或 `Task.run()` 的初始化阶段引入一个**轻量级的图优化 Pass**。这将是我们连接“深度学习框架”与“CPF编译器”的桥梁。

具体实施方案如下：

1. **静态分析与内存规划（Memory Planning Pass）**： 这是为了响应 FAQ 中关于“内存池”的建议。在训练开始前，`Task` 会虚拟执行一次前向和反向过程（不分配实际内存，只推导 Shape）。
   - **产出**：生成一张**内存生命周期表**。
   - **作用**：计算出所需的总显存大小，向 GPU 申请一大块 `Memory Arena`。随后，所有的 `into` 型算子 的输出张量，不再是 `new Tensor`，而是直接指向这块大内存中的特定偏移量（Offset）。这将彻底消除训练循环中的 `cudaMalloc` 开销。
2. **算子融合 Pass（Fusion Pass）**： 我们不需要用户手动修改网络结构去使用 `FusedCBR`。
   - **逻辑**：图遍历算法会自动识别 `Conv` -> `BN` -> `ReLU` 的序列。
   - **动作**：自动将这三个 Module 替换为一个 `FusedCBR` Module 实例。
   - **价值**：用户依然写标准的 ResNet 代码，但框架在运行时自动获得了融合算子的性能。
3. **执行计划展平（Execution Plan Flattening）**： 为了减少 Python 侧（或者 C++ 侧虚函数调用）的 Overhead，优化器可以将整个计算图展平为一个线性的 `OpList`。`Task` 在运行时只需在一个紧凑的循环中依次调用底层 C++ 函数，极大降低调度开销。

这个“中间抽象”不需要设计得像 TVM Relay 那么复杂，它目前只需要是一个优化过的**操作序列（Sequence of Operations）**。这也为未来将模型导出给 **CPF编译器** 提供了天然的接口——CPF 只需要读取这个优化后的操作序列即可进行硬件指令生成。

## 问题11

**问：您的AMP实现方案里，只采用BF16。为什么不采用FP16呢？是否打算添加对FP16的支持，作为备选方案？  **

答：我们演示AMP的硬件平台仅限A10和RTX5090。在A10上，FP16和BF16的算力都是125TFLOPS。在RTX 5090上，网传的FP16和BF16的算力都是209.5TFLOPS。这说明至少在较先进的GPU上，BF16的支持与FP16是并重的。而众所周知，BF16的指数位更多，更能避免溢出，因此BF16在训练（尤其是梯度与累加）上更稳定，不易溢出/下溢，因此现代训练框架常默认用 BF16 以减少调整（例如减少或免除 loss scaling）。此外，BF16与FP32之间的转换非常简单快速，这是其明显优势。而且，oneDNN 将 **BF16 作为一等公民数据类型**，使用BF16在整个流程上没有阻碍。

## 问题12

**问：您认为技术觉醒3框架的开发当中，最主要的瓶颈是什么？  **

答：有了湍流的经验，我们对运算的正确性完全有信心；有了技术觉醒2的经验，我们对单卡的FP32的性能也基本不担心。几个关键的改动中，BF16、NHWC我认为都是改在了最正确的方向上，而且修改难度不大，实现起来胸有成竹，基本看得到是有利无弊；几个关键的扩展中，我们选择的都是相当成熟而且优越的第三方库，风险不大。目前技术觉醒3的开发中，我认为最具挑战的是以下几项：**图优化、内存池、多GPU互联、图像预处理、FI-0420开发**。这些都是我被问得最多的，也是技术细节最复杂、实现工作量最大、实现起来不确定性最大的。



# 【八、采纳的意见建议（实时更新）】

## 建议1

**量化训练的数值稳定性保障**  

QAT对BF16/FP32混合训练敏感，需特别注意：  

- 在`QATQuantizer`中实现**动态范围监控**（如记录激活值分布），防止梯度爆炸。  
- 参考TensorRT的QAT实现，为BN层添加**量化感知前向传播**（如用模拟量化后的输入计算均值/方差）。

## 建议2

**性能基准需纳入真实场景指标**  

除FLOPS外，建议增加：  

- **内存带宽利用率**（尤其针对NHWC布局的GPU优化）。  
- **通信延迟**（多卡数据并行时的NCCL/自定义通信库开销）。  
- 使用**MLPerf Training v3.1**的ResNet-50测试套件作为对比基准。

## 建议5

#### 建议在 `Task` 或 `Model` 初始化阶段增加“自动图优化Pass”

文档设计了 `FusedCB`、`FusedCBR` 等融合层类（2.5.15-18，3.2.19-22）。如果要求用户手动在构建模型时直接实例化这些 `Fused` 类，会降低“易用性”和代码的通用性（比如 ResNet 定义代码变复杂）。
**建议：** 既然是**静态图**框架，建议保持用户定义模型时使用标准的 `Conv`, `BN`, `ReLU`。在 `Task.compile()` 或模型初始化阶段，编写一个简单的**Graph Pass（图遍历优化器）**，扫描 `Sequential` 容器，自动识别“Conv-BN-ReLU”的模式，并将其替换为 `FusedCBR` 实例。
**理由：** 这样既保留了高性能（算子融合），又保证了对开发者的友好性（低代码、标准写法），符合“以开发者为中心”的原则。

## 建议6

#### 建议加强数据加载流水线的“多级缓冲”设计

文档提到图像预处理使用 Simd 库在 CPU 上多线程进行（2.1.30）。在 ResNet-50 这种计算密集型较小、IO吞吐要求极高的模型上，如果 GPU 跑得太快（特别是使用了 `_into` 优化后），CPU 的数据预处理往往会成为瓶颈。
**建议：** 在 `Task` 或 `OneEpoch` 循环中，设计一个明确的 **Producer-Consumer（生产者-消费者）** 模型。C++ 端应开启一个独立的预取线程，维护一个 `Batch` 队列，确保 GPU 在计算当前 Batch 时，下一个 Batch 的数据已经完成 `Simd` 处理并完成了 `HostToDevice` 的拷贝（Pinned Memory/页锁定内存）。
**理由：** 仅有多线程预处理是不够的，还要利用 CUDA Stream 实现数据传输与计算的 Overlap（重叠），否则无法在性能上超越 PyTorch。

## 建议7

**关于测试和验证策略**
文档提到要与PyTorch对齐，误差≤1e-5。但对于BF16和INT8等低精度类型，这个误差阈值可能需要调整。建议为不同精度类型设定不同的验证阈值。

文档要求"与PyTorch误差≤1e-5"，但这对不同数据类型需要区分对待：

| 数据类型 |  建议误差阈值  | 理由                                      |
| :------: | :------------: | :---------------------------------------- |
|   FP32   |      1e-5      | 标准浮点精度                              |
|   BF16   |  1e-2 ~ 1e-3   | 仅7位尾数，精度本身有限                   |
|   INT8   | 量化误差范围内 | 需单独定义量化精度指标(如余弦相似度>0.99) |

建议在测试框架中建立**精度敏感的验证机制**，避免后期调试时在"是bug还是精度限制"上浪费时间。



# 【九、考虑中的建议（实时更新）】

## 建议1

您在文档中明确了这是一个静态图框架，并强调了`into`型方法在避免内存反复分配上的巨大成功。这是非常正确的方向。我的建议是，将这两点系统性地结合起来，在`Task`执行前引入一个“预计算/预分配”阶段，构建一个与计算图绑定的内存池。

*   **具体做法**：
    1.  **图分析阶段**：在`task.run()`被调用时，不要立即执行，而是先遍历整个模型（`Module`树）和训练流程，分析出所有中间张量（前向传播的激活、反向传播的梯度）的生命周期和所需的最大尺寸。
    2.  **内存规划**：根据分析结果，计算出完成一个完整`step`（forward + backward + update）所需的总内存大小。然后，向当前`Device`（如`CudaDevice`）一次性申请一块大的内存（Memory Arena）。
    3.  **地址映射**：为计算图中的每一个中间张量预先分配好在这块大内存中的偏移地址。这意味着，在实际计算发生前，每个操作的输出张量（即`into`型方法中的`result`参数）应该指向哪里，就已经确定了。
    4.  **执行阶段**：`Trainer`在执行计算时，不再动态创建结果张量，而是从这个预先规划好的“地址簿”中取出相应的`Tensor`（它只是一个指向内存池特定区域的句柄），并将其作为`into`型方法的输出目标。

*   **价值与可行性**：
    *   **极致性能**：这能从根本上消除训练循环中的大部分内存分配和释放操作，将`into`型方法的优势从“算子级”提升到“图级”，是实现“超越PyTorch”目标的关键一步。
    *   **契合设计**：这完美契合了您的静态图和`into`型方法设计，是现有架构的逻辑延伸，而非重构。
    *   **可行性高**：由于您已将功能聚焦于CNN，模型结构相对固定，图分析的复杂度是可控的。

## 建议4

#### 任务(Task)执行前增加「Graph Memory Planning Pass」，实现算子级内存优化

文档第7节和“考虑中的建议1”已奠定思路，可抽象为**“静态图预内存规划器”**：

- **执行时机：** 在 `Task.run()` 首次调用前执行。
- **功能：**
  1. 遍历模型的前向与反向图；
  2. 分析所有张量的**生命周期**（创建-最后使用）；
  3. 分配统一「内存池 Memory Arena」，并将所有中间结果映射到预先偏移；
  4. 与 `into()` 函数天然集成。
- **预期收益：**
  - 完全消除训练时的内存分配操作；
  - 对ResNet-50这类静态拓扑网络，训练速度可额外提升 **15~25%**；
  - 为未来FPGA部署提供图态内存优化模板。

> 🔧 若短期内尚不实现完整内存池，可先用`cudaMallocAsync`/`cudaFreeAsync`封装，获得CUDA侧性能收益。

## 建议6

#### 在CUDA后端接入 cuDNN Graph API (v8/v9) 以实现真正的“算子融合”

文档中定义了 `FusedCBR` (Conv+BN+ReLU) 等层。如果在实现时只是在 C++ 层面依次调用 `cudnnConvolutionForward` -> `bn_fwd` -> `relu_fwd`，那么这只能节省 C++ 调用的开销，**无法节省 GPU 显存带宽**，因为中间数据还是要读写三次显存。

*   **具体建议**：
    NVIDIA 的 cuDNN v8/v9 引入了 **Backend API (Operation Graph)**。你们需要在 `Device/Backend` 层封装这个 API。
    当用户使用 `FusedCBR` 时，你们应该构建一个 cuDNN Graph，让 NVIDIA 驱动在 GPU 内部生成一个融合的 Kernel。这样数据在 Conv 计算后直接在寄存器或 L2 Cache 中进行 BN 和 ReLU，最后只写回一次显存。
*   **收益**：
    这是 ResNet-50 训练速度能否超越 PyTorch（PyTorch 也在深度优化这一点）的关键技术点。

## 建议7

#### 强化数据加载的“流水线并行 (Pipelining)”与“页锁定内存 (Pinned Memory)”

文档提到了使用 Simd 库在 CPU 做预处理。但在高性能训练中（尤其是使用了上述内存优化后），GPU 计算会非常快，CPU 数据加载极易成为瓶颈。

*   **具体建议**：
    在 `MnistDataset` 或通用的 `DataLoader` 实现中，必须强制通过 `cudaHostAlloc` 分配**页锁定内存（Pinned Memory）**来存放预处理后的 Batch 数据。
    实现一个双缓冲（Double Buffering）队列：当 GPU 在计算 Batch $N$ 时，CPU 线程正在预处理 Batch $N+1$ 并通过 `cudaMemcpyAsync` 将其**异步流式传输**到 GPU。
*   **收益**：
    掩盖 Host-to-Device 的传输延迟。如果没有这一步，无论 GPU 算子多快，显卡都会因为等待数据而空转（Starvation）。
