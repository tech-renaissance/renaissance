# 技术觉醒深度学习框架设计方案

# 【一、前言】

我们的团队正在开发一款轻量级、高性能、多后端、可重构、跨平台的自研深度学习框架，名为**技术觉醒**（tech-renaissance）。

首先，这个框架并不是我们的第一个框架，也不是第一个版本。几年前我们开发过轻量级的深度学习框架“湍流”，具备最基本的深度学习框架功能，但只支持CPU，不支持其他后端，且优化较差。今年开始开发的技术觉醒框架（最新版本是到V2.4.6），此前已取得阶段性成果，实现了基本模块和核心算子，FP32的性能略优于PyTorch。因此我们已经具备一定的经验。

但是，技术觉醒框架的第2代的基本设计存在一些问题。主要在于此前的考虑是将其作为一个小型的试验项目，对可扩展性考虑不足，也没有大胆设想跟主流的开源深度学习框架进行性能上的竞争，整体设计偏保守和狭隘。主要局限性体现在：第一，只规划了FP32和INT8、INT32，没有考虑AMP的实现。第二，只考虑了单卡训练，没有考虑多卡的数据并行的实现。第三，张量的数据布局采用传统的NCHW，而当前GPU对NHWC的加速支持更为强大。第四，对于多后端尤其是边缘设备推理的考虑不够充分。第五，第2代框架设计初衷直接定为开源，没有充分考虑闭源的需求。这五点局限性促使我们重新设计技术觉醒框架，而非在原来的基础上进行修改，因为这涉及到数据类型、数据布局、算子库的调整，改动极大。对于AMP、数据并行、NHWC的支持，都是出于训练性能考虑，毕竟在大规模训练中，主流深度学习框架一般会把这些选项都开启，以获得最优性能。如果我们要在性能上跟主流深度学习框架比拼，这几种功能是必须支持的，否则性能上的优势就没有说服力。此外，第2代的技术觉醒框架还存在一个“后端”与“器件”术语混淆的小问题，显得不够专业。

我们现在要设计开发的，是**技术觉醒3**。

需要特别强调的是：第一，我们拥有一定的经验和基础，实现一个性能良好的轻量级自研深度学习框架、甚至在部分指标上比肩主流框架并非痴人说梦。第二，这依然只是一个轻量级深度学习框架的初期版本，它并非产品，它当前仅用于演示、教育、交流、学习、研究，它并不追求功能的面面俱到，只求在专注的领域上实现足以演示的功能和较好的性能。

简单来说，我们的自研深度学习框架当前**只做CV，而且只做CNN，而且只实现最核心最常用的那一部分模块**，但是，**它最好要在ResNet-50的训练和推理上比肩和超越PyTorch**。

因为有了此前的开发经验，我们对功能定义、工具选型有了一个相对而言更为完善而且实际的方案。为了节约时间、人力、物力等成本，我们依然会严格控制深度学习框架所需要拥有的功能，避免功能无序扩张、贪大求全而导致多而不精，甚至项目烂尾。

尽管有了前面的经验和强大的第三方库，以及有AI编程的助力，但开发工作量依然非常巨大。下面介绍我们的基本设计思路。



# 【二、基本设计】

## （一）基本属性

1. 这是一个静态图深度学习框架。
2. 本框架当前**只做CV，而且只做CNN，而且只实现最核心最常用的那一部分模块**。
3. 本框架的开发者包括人和AI，整个过程会充分利用AI编程，也会借助AI进行设计和决策。
4. 本框架**以开发者为中心**。
5. 本框架需要适配的操作系统是Windows（主要是Windows11）和Linux（主要是Ubuntu 24.04 LTS，当然，还需要支持边缘设备的Linux系统）。
6. 本框架会用C++开发底层，在功能实现并完成验证后，用PyBind11进行封装，最后提供C++和Python两套API。
7. 本框架不能依赖于其他的任何深度学习框架。但是，它可以使用包括高性能计算库在内的多种第三方库。第三方库尽量使用vcpkg来管理。
8. 由于会用到各种第三方库，本框架采用Python脚本来自动配置依赖。
9. 本框架支持的Tensor的数据类型包括：FP32、BF16、INT32、INT8，总共4种。
10. 本框架采用的数据排列格式是**NHWC**。CPU和GPU上都是。目前越来越多的框架都在强调对 **NHWC (channels last)** 的支持。根据 NVIDIA 的白皮书，ResNet-50 在 A100 上使用 NHWC + TensorCore 可获得 1.3-1.8× 的端到端加速。此外，关于“最符合人类思维的维度顺序”，我问了10个AI，10个AI都回答说是**NHWC**。原生支持NHWC，我们将同时拥有性能和易用性的优势。
11. 本框架的六大板块是：数据（data）、模型（model）、训练器（trainer）、器件（device）、任务（task）、工具（utils）。
12. 本框架所追求的“可重构”主要体现为算子、器件支持的模块化、插件化设计，后期将可以按需选择部件，将框架裁剪成一个更小的子集。举例来说，可以只支持全连接而不支持卷积，可以只支持CPU而不支持GPU，用不上的模块可以不勾选，使得框架更为精简，且去掉多余模块后，框架的其他功能依然不受影响。
13. 本框架的最核心特色之一是**后端解耦**，目的是方便后端开发者进行扩展，尤其方便自研人工智能芯片的开发者。后端解耦的主要表现包括：
    1. Tensor本身不定义任何运算功能，所有运算由其所在的器件执行，需要显式调用器件对象的方法。
    2. 各后端分别独立实现算子。
    3. 每个后端并不需要实现所有的算子，当后端不支持该算子时，会运行时报错。
14. 本框架计划支持的器件包括：**x86 CPU**（必须支持Intel和AMD，但测试主要是在Intel Core i9-14900HX和Intel Xeon Gold 6530）、**NVIDIA GPU**（主要测试A10、RTX 4060、RTX5090）、**摩尔线程GPU**（主要测试MTT S80）、**FPGA**（主要测试Intel SoC）、**ARM**（主要测试树莓派5）。除此以外，本框架还将支持模型导出，导出的模型将会在Intel NCS2、华为Atlas 200i DK A2上进行INT8推理试验。这里我们非常明确，Intel NCS2、华为Atlas 200i DK A2属于专用硬件，没有必要去替厂商开发专用硬件的支持，而是应该直接使用它们的工具进行推理。
15. 本框架将支持模型的8-bit量化，包括PTQ和QAT。量化后的INT8模型将可以运行在x86 CPU、NVIDIA GPU、FPGA、ARM。
16. 自动混合精度（AMP）训练主要使用BF16和FP32。
17. 支持的数据集只需要包括：MNIST、CIFAR-10、ImageNet。
18. 优化框架性能的最关键思路是**避免内存的反复分配**以及**零拷贝**。我们在技术觉醒2中设计了**into型方法**，也就是将运算的结果写入指定的张量而非返回一个新的结果张量（比如void mul_into(const Tensor& a, const Tensor& b, Tensor& result)就是把张量a、b逐元素相乘的结果写入同形张量result），这种方法在实践中取得了接近50%的提速，非常成功，是我们的技术觉醒2框架在运算性能上超越PyTorch的关键。
19. 考虑到时间人力成本，此前的技术觉醒2的规划是“暂不实现内存池”。但在技术觉醒3中，内存池的实现值得重点考虑。
20. 本框架的C++命名空间是tr。Python的包名是renaissance，导入后的默认简称也叫tr。
21. 本框架的主头文件是renaissance.h。测试样例一般只需包含主头文件和必要的标准库头文件，而不包含框架内部的其他头文件。
22. 由于Windows下CUDA的兼容性问题，我们在Windows下使用MSVC作为默认编译器。对于Windows下的与CUDA无关的功能，可使用MSYS2的g++进行编译。而在Linux，默认使用g++。
23. C++采用CMake来构建项目。
24. 出于未来的利益考虑，我们决定在开源问题上暂时**有所保留**。技术觉醒3框架将分为**完整版**和**开源版**。完整版将托管在Gitee的闭源仓库，而开源版将在完整版功能开发验证完成后制作，进行合理裁剪，阉割部分功能，然后公开发布到GitHub和Gitee。
25. 本框架的完整版将在编译成库之后进行Docker分发，通过Docker提供试用。
26. 本框架已有相应的官方域名（技术觉醒.中国、tech-renaissance.cn），且已备案完成，在开发基本完成后，将上线官网，并在官网提供介绍和使用教程等。
27. 本框架的核心演示目标是ResNet-50和MobileNetV2，原则上只要是这两个模型没有的运算，都暂时不需要支持。
28. 本框架需要借助PyTorch来调试，所以至少在早期是需要与PyTorch的功能对齐的，也就是，同样一个操作，我们要得出跟PyTorch一致的结果。
29. 代码风格尽量遵循业界规范（如Google C++ Style Guide），但我要求全部注释使用中文，全部打印输出及异常信息都使用英文，代码里不能包含任何emoji，以免碰到兼容性问题。
30. 图像预处理考虑采用ermig1979的Simd库来实现，运算全部在CPU上进行，要充分利用多线程。
31. 本框架主打易用性，API要简洁易用，并且允许使用YAML进行快速配置，像NVIDIA TAO一样实现低代码的模型训练。
32. “三驾马车”：与本框架一道开发的项目还包括AI编译器**“算力场**”（Compute Power Field，CPF）、基于FPGA的AI芯片“**FI-0420**”，它们主要是用来演示我们的框架对AI芯片研发的助力。但重心依然是在深度学习框架上。
33. 支持模型导出、数据导出（张量导出）和数据集导出。模型导出，我们可以有自己的格式（.mdl），然后可以导出为ONNX（这个非常重要！必须实现）。数据导出，我们有自己的格式（.tsr），然后还要支持导出为.pth（需要安装pytorch，调用脚本转化）、.npy（需要安装numpy，调用脚本转化）。数据集导出，其实就是保存数据集里面所有的样本张量和标签，这是为了统一格式，给用户更好的体验，我们用自己的格式（.dts）。
34. 支持保存checkpoint（.ckpt），支持断点续训，也就是能完全保存训练状态，包括模型结构、模型参数、模型超参数、训练器内保存的动量、训练执行到哪个批次等。
35. 深度学习模型中所有的“层”都属于模块类（Module）的子类，根据我分析的需求，我们只需要实现以下几个层：卷积层（Conv）、全连接层（Linear）、ReLU激活函数（ReLU）、BN1D、BN2D、最大池化（MaxPool）、全局平均池化（GAP）、Flatten。
36. 损失函数只支持交叉熵。
37. 关于ResNet结构里最重要的Shortcut，我的观点是，通过“并行容器”来实现。容器就是决定各个层的连接的结构，它分为“顺序容器”（Sequential）和“并行容器”（Parallel），容器可以装各种不同模块，顺序容器里面各个层都是首位顺序连接，而并行容器里面各个层都是享有相同的输入，然后最后把输出加起来。
38. 需要支持模型结构打印，也就是在控制台输出文本，表明这个模型的结构是什么。
39. 需要支持标签平滑（label smoothing）。
40. 支持三种优化器：SGD with momentum、Adam、AdamW。
41. 支持七种学习率调度器：常数学习率、StepLR、MultiStepLR、指数、余弦退火、带热启动的余弦退火、用户自定义函数的学习率调度器。
42. 支持算子融合。如果卷积层是C、BN层是B、全连接层是L、ReLU层是R，那么咱们的框架支持的算子融合就是4种：CB、CBR、LR、LBR。
43. NVIDIA GPU上一定要使用CUDA，C++开发。而不要使用PyCUDA或者OpenCL之类。
44. GPU的显存，RTX 4060是8GB，摩尔线程MTT S80是16GB，A10是24GB，RTX 5090是32GB。我们用于研发和测试的主要平台是配备两个RTX 5090的服务器。
45. 完成基本开发后，我会借助AI来生成文档和示例程序，确保用户在拿到我的文档和示例程序后，可以通过对AI下达命令，就让AI自动生成能运行于技术觉醒框架的代码。
46. 关于workspace目录管理：我们在项目根目录下设置workspace目录，临时文件统一放在这里。
47. 演示所使用的ResNet相比官方原版有两个改动：一是最大值池化，我们采用标准的kernel=2、stride=2、padding=0，以简化硬件实现；二是最后的全连接层，我们去除bias。

# 【六、开发原则】

## 一、总体原则

1. **以开发者为中心**：优先保证开发效率和可行性，不盲目追求用户友好性；功能不无序扩张。
2. **最小可行产品（MVP）先行**：限制功能，先把最小可行产品实现好，功能如前文所述，非必要不添加。
3. **结果对齐验证**：开发过程中，关键运算的结果要尽量与 PyTorch 输出结果在数值上高度一致（误差 ≤ 1e-5），建立自动化测试体系。
4. **静态图 + 手动求导**：在静态图模式下，整个计算图的内存需求可以在执行前被**完整分析**。不实现自动微分引擎；所有层需手动实现前向与反向传播逻辑。

## 二、代码与工程规范

1. **命名与风格**：
   - C++ 代码遵循 **Google C++ Style Guide**。
   - 所有注释使用 **中文**，所有打印输出（如日志、错误信息）使用 **英文**。而且，**注释和输出当中都不能包含emoji**。
   - 主头文件为 `renaissance.h`，命名空间为 `tr`；Python 包名为 `renaissance`，常用别名为 `tr`。除非特别要求，否则切勿在命名空间tr之下再添加其他命名空间，以免过度复杂化。
   - 头文件使用`#pragma once`防止重复包含。
   - 括号：左括号不换行，右括号独占一行。
   - 注释：类/函数——**使用Doxygen格式**（如`/** ... */`），包含功能描述、参数说明；复杂逻辑——行内注释解释（中文）。
   - 版本号：V+三个整数。比如：V3.1.4。不使用多余的前置0。
2. **文件组织**：
   - 参考材料中给出的目录结构（`include/renaissance/`、`src/`、`tests/` 等）。
   - 头文件与源文件按功能要素（数据、器件、模型、训练器、任务、工具）分目录组织，结构镜像。
   - 测试样例产生的临时文件统一放在根目录下的workspace文件夹中，且通过WORKSPACE_PATH宏来引用。该文件夹会自动创建。
   - 所有测试样例产生的临时文件，除特别声明以外，都应该在测试结束时删除
3. **构建系统**：
   - 使用 **CMake** 作为唯一构建工具。
   - 通过 `option()` 实现“可重构”编译（如 `TR_BUILD_CONV_MODULE`）。
   - 分层CMake结构：根`CMakeLists.txt`管理子目录，子目录（如`src/data/CMakeLists.txt`）负责局部构建。

## 三、核心架构规则

1. **数据类型**：
   - 张量里的数据类型DType只支持4种：FP32、BF16、INT32、INT8
2. **张量与内存模型**：
   - 所有张量的维度上限为 **4 维**
   - 采用 **句柄式内存管理**：`Tensor` → `Storage` → `Device`。
3. **器件抽象**：
   - 定义 `Device` 基类，派生 `CpuDevice`、`CudaDevice` 等。
   - **尽量调用 cuDNN/cuBLAS**，不要手写 CUDA Kernel。
   - 器件通过 `DeivceManager`（单例）全局访问。
4. **模块与算子**：
   - 所有层继承自 `Module`，不单独设立“算子类”。
   - 支持的层有限制，详见上文。
   - 卷积核限制为 **方形**，且仅支持 **1×1、3×3、5×5、7×7**，输出通道 ≤ 2048。
5. **算子融合**：
   - 支持 4 种融合：CB、CBR、LR、LBR。
6. **任务（Task）机制**：
   - 支持原始的Forward-Backward写法，也支持通过Task机制简化训练推理工作流。
7. **量化支持**：
   - 仅支持 **INT8**，采用 **对称量化 + 逐通道**。
   - 实现 PTQ 与 QAT。

## 四、可重构与可扩展性

1. **编译时裁剪**：
   - 通过 CMake 选项控制模块是否编译（如 `TR_BUILD_CONV_MODULE`）。
   - 源文件头部需注明依赖、功能、版本，便于脚本生成 CMakeLists。
2. **错误与日志**：
   - 使用自定义异常类 `TRException`。
   - 日志系统基于自己的实现，而非第三方库。支持 DEBUG/INFO/WARN/ERROR 级别。
3. **序列化与导出**：
   - 支持导出 .onnx、.mdl、.tsr、.pth、.npy、.json、.dts。

## 五、测试与开发流程

1. **测试体系**：
   - **单元测试**：每个核心类（Tensor、Conv、SGD 等）必须有对应测试，文件命名`test_*.cpp`。
   - **集成测试**：必须包含 `LeNet+MNIST` 和 `ResNet+ImageNet` 端到端训练测试。
   - 不要使用Catch2等第三方的测试框架，以减少依赖。
2. **Fail-Fast校验**：
   - 在构造函数或API入口校验约束：
     - 卷积核尺寸：仅允许1×1/3×3/5×5/7×7，否则抛出`ShapeError`。
     - 张量维度：≤4维。
     - 输出通道数：超过2048时报错。
   - 错误信息：明确指出违反的规则（如"Conv kernel size must be 1, 3, 5, or 7, got 4"）。
3. **开发工具链**：
   - 提供 `configure.py` 脚本，一键完成环境配置、依赖找齐。
   - 可使用 Doxygen 生成 API 文档，注释需符合规范（“文档即代码”）。
4. **异常处理规范**：
   - 统一使用`TRException`（继承`std::exception`），包含错误位置（文件/行号）。
   - 关键路径（如内存分配、设备操作）必须检查返回值，失败时抛出异常。
   - 禁止使用`assert`，替换为运行时校验。
