# 【破解NUMA架构下的多线程谜题】



我们的项目需要在Linux服务器上用多线程实现一个重要的数据处理功能，但是由于NUMA架构的特殊性，此前频频碰到同步超时的问题。大致情况，就是由于NUMA架构下的延迟可达10us量级，各线程在同步时无法确认对方状态，导致同步超时卡死。现在我们需要用C++17写一个原型程序，来验证何种方式能够稳定地避免这个问题。

首先，这个程序要用C++17来写，虽然主要是在Linux上测试，但要兼顾跨平台（代码最终需要在Linux和Windows都能运行）。

这个程序的名称就叫test_thread，运行的命令应该是：

```shell
./test_thread --delay1 3 --delay2 1000 --worker 12 --iteration 2000
```

delay1是预处理的延迟，delay2是传输的延迟，worker是预处理的线程数，iteration是迭代次数。

数组a和数组b代表预处理完后、需要传输的数据。长度都为512。

有worker个线程，在程序结束之前都不join。它们模拟的是预处理的过程，这个用时是1.0×delay1到2.0×delay1之间（随机，单位毫秒），用sleep。sleep完成后，就往一个长度为512的数组a里面写入一个随机数，rand()的结果即可。写入的位置，取决于线程ID。0号线程就写0号索引的位置，1号线程就写1号索引的位置，如果总共12个线程，那么11号线程就写11号索引的位置，但12号索引是属于0号线程的。如果数组a里面已经没有该线程可写的位置了，该线程就停下，并且查询数组b是否可写。当a被写满，a就进入异步传输状态，不可写入，在经过1.0×delay2到2.0×delay2之间的时间（随机，单位毫秒）后，数组清零（代表异步传输完成），变为可写。数组a在传输的过程中，只要b可写，所有预处理线程就去向b写入。如果b还在传输中，不可写，那预处理线程就等待。写完a后写b，写完b后写a，前提是这个数组是可写的状态。iteration就是，每个线程各要写入iteration个元素。所有的线程都写完了它的iteration个元素后，数组a或b进行最后一次传输，程序结束。

现在关键就在于这个状态的切换。数组怎么判断写满了？异步传输谁来触发？如何等待另一个数组变成可写？

请分析我们的需求，设计这样的程序，返回完整的代码。一定要使用安全、高效的写法，彻底避免同步超时问题，并且避免严重的锁竞争影响速率。如果发生超时（5s），应该报错退出。如果全部iteration都执行完毕没有碰到问题，应该报告整个程序总共运行了多长时间，然后退出。
