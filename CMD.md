# 注释与代码排版规范化纲领

---

## 一、目的

我们的框架核心功能已通过全部测试，准备开源。在此之前，需要对代码进行规范化，目的有三：

1. 让用户和开发者感受到代码的专业性；
2. 让用户和开发者能容易理解代码；
3. 完善而规范的注释便于后续撰写文档和 API 说明。

---

## 二、修改范围

`include/`、`src/`、`tests/` 目录下的所有 `.h`、`.hpp`、`.cpp`、`.cu` 文件。具体范围听从最新指示。

**排除项**：第三方代码（如 `node.hpp` 的 fkyaml 库、`Simd/` 目录等）不纳入本次修改范围。

---

## 三、核心原则

**所有修改不得改变代码的实际逻辑。** 只能修改注释和代码排版，不能修改任何参数、调试打印信息、变量名称、函数名称、类名、命名空间名、条件判断、常量。一个简单的判断标准：修改后编译产物应与修改前完全一致。

### 3.1 修改前必须理解代码

你必须在深刻理解代码（包括头文件和源文件）的基础上，再修改其注释和格式。不能在不了解、或理解有误、或与现有注释冲突的情况下随意修改。

### 3.2 模棱两可则不动

如果碰到实在无法判断是否应该修改的地方，不要修改。

### 3.3 严禁脚本批量替换

禁止使用脚本或 shell 命令批量替换。必须精确匹配修改，避免批量操作引发未知错误。

---

## 四、文件头规范

每个文件头至少包含以下信息：

```c++
/**
 * @file 文件名
 * @brief 一句话用途说明
 * @version 4.20.1
 * @date 2026-06-28
 * @author 技术觉醒团队
 * @note 所属系列: xxx
 */
```

### 4.1 各字段说明

| 字段 | 写法 |
|------|------|
| `@file` | 带文件名，如 `@file tensor.h` |
| `@brief` | 一句话概括本文件提供了什么 |
| `@version` | 一律 `4.20.1` |
| `@date` | 一律 `2026-06-28` |
| `@author` | 一律 `技术觉醒团队`（注意是中文） |
| `@note 所属系列` | 按文件所在子目录填写（见下表） |
| `@note 依赖项` | 可选。列出本文件直接依赖的关键头文件。若无需填写，则整个字段不出现，不要写空字段 |

如果有必要，也可添加 `@details` 等扩展字段。

### 4.2 所属系列取值

按文件所在子目录填写，沿用项目现有约定：

| 目录 | 所属系列 |
|------|----------|
| `include/renaissance/core/`、`src/core/` | `core` |
| `include/renaissance/tensor/`、`src/tensor/` | `tensor` |
| `include/renaissance/task/`、`src/task/` | `task` |
| `include/renaissance/graph/`、`src/graph/` | `graph` |
| `include/renaissance/algo/`、`src/algo/` | `algo` |
| `include/renaissance/data/`、`src/data/` | `data` |
| `include/renaissance/backend/`（非 ops 子目录） | `backend` |
| `src/backend/ops/dtensor/` | `backend/ops/dtensor` |
| `src/backend/ops/range/` | `backend/ops/range` |
| `tests/xxx/` | `tests/xxx` |

### 4.3 缺失文件头的处理

对缺少文件头的文件，按模板补齐。

## 五、Doxygen 注释风格

### 5.1 两种 Doxygen 风格的分工

项目同时使用 `/** */` 和 `///` 两种风格，各有适用场景，不可混用：

#### A. `/** */` 多行风格 —— 正式文档

适用于：**文件头、类声明、结构体、函数声明、枚举定义**。

```c++
/**
 * @brief 模板方法：以指定类型指针访问数据
 * @tparam T 目标类型
 * @return T* 类型化指针
 */
template<typename T>
T* data() noexcept { ... }
```

对简单的存取函数，可简化为单行：

```c++
/** @brief 获取形状 */
const Shape& shape() const noexcept { return shape_; }
```

#### B. `///` 和 `///<` 单行风格 —— 简洁标注

适用于：**枚举值、结构体成员、union 成员、简单成员变量**。

```c++
enum class GraphId : uint8_t {
    TRANSFER_A,        ///< H2D 异步传输 A 区（双缓冲前半）
    TRANSFER_B,        ///< H2D 异步传输 B 区（双缓冲后半，异步重叠）
    ...
};

float momentum = 0.9f;    ///< 动量系数
bool  nesterov = false;   ///< 是否使用 Nesterov 动量
```

#### C. `//` 普通注释 —— 实现细节

适用于：**函数体内的局部逻辑说明、临时标记**。

```c++
int dims[4] = {1, 1, 1, 1};  // NHWC格式：dims[0]=N, dims[1]=H, dims[2]=W, dims[3]=C
```

### 5.2 Doxygen 标签体系

| 标签 | 适用场景 | 要求 |
|------|----------|------|
| `@brief` | 文件头、类、函数、枚举、结构体的一行摘要 | **必须** |
| `@param` | 每个函数参数 | 有参数时**必须** |
| `@return` | 有返回值的函数 | 有返回值时**必须** |
| `@tparam` | 模板参数 | 有模板参数时**必须** |
| `@throws` | 可能抛出异常的函数 | 按需使用 |
| `@note` | 补充说明、设计决策、换算关系、使用提示 | 按需使用 |
| `@warning` | 性能敏感、禁止行为、重要前置条件 | 按需使用 |
| `@details` | 需要展开说明的实现细节 | 按需使用 |
| `@deprecated` | 废弃但保留兼容的 API | 按需使用 |

- `void` 函数和简单存取器可省略 `@return`。
- 长说明既可以用 `@details`，也可以直接写在 `@brief` 后面的裸段落里，同一文件内保持一致即可。
- `@brief` 句末是否加句号不强求，同一文件内保持一致即可。

### 5.3 格式细节

- 多行 `/** */` 注释中每行开头的 `*` 对齐。
- 可配合 `@code` / `@endcode` 嵌入代码示例，嵌入的代码保留其原有缩进和换行。

### 5.4 注释语言

- 注释以中文为主，**不使用 Emoji**。
- 行业标准技术术语保留英文或缩写：`CUDA`、`GPU`、`CPU`、`H2D`、`D2H`、`NHWC`、`FP16`、`FP32`、`stride`、`padding`、`warp`、`block`、`grid` 等。
- 专有名词保持原样：`cuDNN`、`cuBLAS`、`NCCL`、`mimalloc`、`libcurl` 等。
- 不适合用中文表述的内容，不强求翻译。

## 六、必须清理的注释内容

### 6.1 计划代号

删除形如 `[EXY2]`、`[XXX]` 等内部计划代号，保留后续技术说明。

> 例：`[EXY2] 让 Compiler 的 I_A_DATA 分配与 ShapeId 跟随 ArchPlan` → 删除 `[EXY2]`。

### 6.2 人名引用

删除所有注释中的人名（如"姜总工"、"专家 SN"、"专家 CG"等），将表述改为客观技术描述。

> 例：`（姜总工的要求：最多相差1）` → `样本数最多相差 1`

### 6.3 内部临时文件引用

删除"根据 XXX.md"、"根据 XXX 计划"等指向临时内部文件的表述，仅保留技术内容本身。

> 例：`根据list.md的实际测量数据（2026-02-22统计）` → `根据实际测量数据`

### 6.4 TODO / FIXME / XXX / HACK / BROKEN

| 标记 | 处理规则 |
|------|----------|
| `TODO` | 已实现则删除；未实现则保留，可补充说明"未实现原因"或"预期实现条件" |
| `FIXME` | 保留，但把表述改专业，去除情绪化内容和人名 |
| `XXX` / `HACK` | 保留，但应补充说明"为什么必须这样写"，不要只留裸标记 |
| `BROKEN` | 保留并说明当前状态 |

注意：API 头文件中"当前仅提供 API 支持，具体实现留为 TODO"的说明不应删除，那是接口状态说明，不是临时标记。

### 6.5 过时或与代码不符的注释

注释应与代码实际逻辑相符。如果看到过时的注释或明显与代码逻辑不符的注释，应修改注释。

### 6.6 武断或不专业的表述

如果注释中存在武断的、不专业的表述，应修改或删除。对于警告性、提醒性的语句，应保留其技术含义，但如果表述过度夸大、语言浮夸（如连续大量感叹号），应修改为专业表述。

### 6.7 函数体内的技术长注释

函数体内部有详细技术说明的注释（如 `types.h` 中 `Shape` 构造函数的 MSVC 优化问题说明），**技术内容保留**，但：

- 删除内部的 `@date` 行、`@bug` 等 Doxygen 标签（这些应放在函数头注释中）。
- 删除情绪化标题（如 `【重要编译器优化问题】`），改为中性标题（如 `// 重要：MSVC /O2 优化问题`）。
- 删除"技术觉醒团队调试发现并修复"等内部标记。

### 6.8 被注释掉的代码

被注释掉的代码段按以下原则处理：

- **临时调试代码**（如被注释掉的 `fprintf(stderr, ...)`、`TR_LOG_INFO(...)`、临时断言等）：若明显无保留价值，可删除。
- **条件编译备用实现、有设计参考价值的旧逻辑、暂时禁用但可能重新启用的代码**：保留，并可补充说明保留原因。
- 删除被注释代码**不改变实际编译逻辑**，符合本文核心原则。

### 6.9 保留有价值的信息

不可以破坏注释中那些虽然这里没有提到、但是明显有价值的信息。

## 七、排版与格式

排版基本符合 Google C++ Style 规范（名称相关的不在本次修改范围）。以下为项目已有惯例，按此执行：

### 7.1 缩进

每层 4 个空格，不使用 tab。

### 7.2 大括号

函数、类、命名空间的左大括号**不换行**（K&R 风格，项目现有惯例）。

### 7.3 空行

- 函数定义之间保留 1 个空行。
- 文件内不同逻辑段落之间可保留 1 个空行。
- 不随意增删逻辑块内的空行。

### 7.4 操作符空格

`a + b` 风格（项目现有惯例）。

### 7.5 分节注释

文件内的大模块分隔推荐使用 `// ========...` 或 `// --------...` 风格。**同一文件内保持统一**，不强制跨文件使用相同符号。

### 7.6 命名空间结束注释

命名空间结束处补齐 `} // namespace xxx` 注释。项目已有此惯例，补齐缺失的即可。

### 7.7 行宽

不设硬性上限。对明显可拆分的超长行可在语义自然处换行。

### 7.8 不强制的规定

以下排版细节不做统一要求，维持现状：

- 行宽上限（80/100/120）。
- 长参数列表的换行对齐方式。
- 运算符空格、逗号后空格等微观格式。
- 文件末尾是否保留空行。

## 八、禁止修改的边界

以下内容**严禁修改**，否则会改变编译产物或引发问题：

### 8.1 头文件包含顺序

严禁修改 `#include` 顺序。当前写法编译通过且运行正常，修改可能造成问题。

### 8.2 宏定义

涉及宏定义的，不要改变其缩进或换行。`#ifdef`、`#endif` 等后面的行尾注释可在保持逻辑清晰的前提下整理位置。

### 8.3 编码与换行符

本次**不修改**文件编码（UTF-8 BOM）和换行符（CRLF/LF），维持现状。换行符统一问题留待后续处理。

### 8.4 已注释掉的代码

已注释掉的代码段按第 6.8 节原则处理：删除明显无保留价值的临时调试代码，保留有参考价值或可能重新启用的代码。

### 8.5 第三方代码

第三方代码（如 `node.hpp` 的 fkyaml 库、`Simd/` 目录）不纳入本次修改。

## 九、执行方式

按**板块（目录）分阶段**执行。每个目录作为一个批次，按字典序推进。例如：

1. `include/renaissance/core/`
2. `src/core/`
3. `src/graph/`
4. 其他目录依此类推

每个阶段内：

1. **一步到位彻底改好**：对该目录下的每个文件，一次性完成文件头统一、注释清理、Doxygen 规范化、排版整理等全部内容修改。
2. **完成该目录全部文件后，暂停并编译验证**：对照 `build/` 目录与 `build_legacy/` 目录的产物哈希值，确保一致后再进入下一阶段。
3. **准备样板**：可在首个阶段挑选 2-3 个已有较好注释的文件（如 `tensor.h`、`logger.h`、`downloader.h`）作为参考标杆，确定本阶段风格后再批量处理同目录其余文件。

## 十、验收标准

修改后增量编译，对照 `build/` 目录与 `build_legacy/` 目录下对应编译产物的哈希值（`build_legacy` 为修改前的 build 产物，目录结构完全对应）。如果修改正确，哈希值应完全一致。若不一致，优先检查：

- 是否误改了代码逻辑、宏定义换行。
- 是否意外修改了文件编码或换行符。
- 是否误删或误改了非注释的代码内容。
