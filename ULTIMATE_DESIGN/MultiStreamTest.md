# Conv+BN+ReLU 反向传播多流策略测试报告

**版本**: 1.0.0  
**日期**: 2026-05-16  
**测试平台**: NVIDIA A100-SXM4-80GB, CUDA 13.1, cuDNN 9.17, cuDNN Frontend 1.17  
**算子配置**: Heuristic Mode B, CUDA Graph 模式  

---

## 一、测试目的

Conv+BN+ReLU 反向传播包含三个子操作：

- **BN Backward**: 输入 dy, x, scale, mean, inv_variance, bitmask → 输出 dx, dscale, dbias
- **DGrad**: 输入 grad_output (dx), weight → 输出 grad_input
- **WGrad**: 输入 grad_output (dx), input → 输出 grad_weight

我们实现了三种流调度策略，对比其性能差异：

| 策略 | 文件 | 拓扑结构 |
|------|------|---------|
| **三流并行** | `cbr_bwd_fp16.cpp` | S1[BN BWD] → Fork S2[DGrad] ∥ S3[WGrad] → Join S1 |
| **三流顺序** | `cbr_bwd_fp16_seq.cpp` | S1[BN BWD] → S3[WGrad] → S2[DGrad] → S1 |
| **单流** | `cbr_bwd_fp16_single.cpp` | S1[BN BWD → WGrad → DGrad] |

核心研究问题：**三流顺序能否在不增加工程复杂度的情况下，获得与三流并行等价的性能？**

## 三、测试方法

**独立重复实验**: 进行了 6 次完全独立的批量测试，每次包含全部 8 组配置 × 3 种策略 = 24 个数据点。

每个数据点内部执行流程：
1. 50 次传统模式 warmup
2. CUDA Graph 捕获
3. 50 次 Graph 模式 warmup
4. 500 次 Graph 模式 benchmark，取平均耗时

**总计 6 × 24 = 144 个独立数据点**，所有原始数据见项目根目录 `data.txt`。

## 四、测试结果

### 4.1 全局统计（144 个数据点，不分形状）

| 策略 | 平均值 (ms) | 中位数 (ms) | 标准差 | Min | Max |
|------|-----------|-----------|--------|-----|-----|
| **三流并行** | 1.4124 | 1.1775 | 0.5957 | 0.6945 | 2.2441 |
| **三流顺序** | **1.3989** | **1.1480** | 0.5799 | 0.7169 | 2.2648 |
| 单流 | 1.4011 | 1.1530 | 0.5835 | 0.7014 | 2.2830 |

**三流顺序在平均值和中位数两个核心指标上均优于三流并行和单流。**

> **注意**: 不分 shape 混合统计时，0.7ms 和 2.3ms 的数据点权重相同，相互抵消。全局均值差异仅为 0.16%，远小于标准差（~0.58），说明混合统计会抹平特定 shape 下的显著差异。**分 Group 分析（4.2）才是决策依据。**

### 4.2 分 Group 统计（6 次 Run 平均）

| 组 | 配置 | 三流并行 | 三流顺序 | 单流 | Group 最优 |
|----|------|---------|---------|------|-----------|
| 1 | 14×14, 256→1024, 1×1 | **1.1383** | 1.1540 | 1.1552 | 并行 |
| 2 | 7×7, 512→512, 3×3 | **1.0074** | 1.0208 | 1.0141 | 并行 |
| 3 | 56×56, 256→64, 1×1 | 2.1764 | **2.0514** | 2.0508 | 顺序/单流 |
| 4 | 28×28, 128→512, 1×1 | **2.0129** | 2.0239 | 2.0274 | 并行 |
| 5 | 7×7, 512→2048, 1×1 | **0.7930** | 0.8046 | 0.8060 | 并行 |
| 6 | 56×56, 64→64, 3×3 | **2.2436** | 2.2640 | 2.2789 | 并行 |
| 7 | 28×28, 512→128, 1×1 | 1.2064 | **1.1423** | 1.1474 | 顺序 |
| 8 | 14×14, 1024→256, 1×1 | **0.7211** | 0.7304 | 0.7288 | 并行 |

### 4.3 关键观察

1. **三流并行 vs 三流顺序是同一策略的噪声级差异**。除去 Group 7，其余 7 组两者的差异均在 1-2% 以内，且互有胜负。Group 7 是唯一的异常（顺序领先 5.3%），这一组的数据一致性高（6 次 Run 的标准差极小），可能是该 shape 下 DGrad/WGrad 的 SM 争抢导致的并行惩罚。

2. **三流策略 vs 单流有确定性优势**。将并行和顺序视为同一策略族（三流），取每次 Run 中三流的最优值与单流比较，在全部 48 次 Run-Group 组合中存在显著优势：多流最佳 vs 单流赢 39 次、输 9 次 → 胜率 **81.2%**。其中三流并行对单流赢 29 次、三流顺序对单流赢 28 次。9 次单流获胜中有 6 次差距 < 0.5%，属平局级别。

3. **空间尺寸越小，三流优势越明显**。7×7 场景下三流最优平均领先单流约 1.1%（Group 2: 0.7%, Group 5: 1.6%），56×56 场景下降至约 0.8%。小 feature map 的 kernel 线程块数少，单 kernel 无法吃满 SM，三流的额外并发可以填充空闲计算单元。

4. **Group 3（56×56, 256→64, 1×1）是三流并行的 consistent 弱项**。6 轮测试中三流并行全部最慢，比单流慢 5.9%~6.4%。此 Group 特征为高输入通道→低输出通道的 1×1 卷积，DGrad 和 WGrad 的计算量相近但内存访问模式差异巨大（DGrad 输出大张量 256×56×56，WGrad 输出小张量 64×256×1×1），并发时 DRAM 带宽争抢严重，反而比顺序执行更慢。这进一步证实了**三流并行的并发收益高度依赖 shape**，而三流顺序无此弱点。

### 4.4 统计显著性说明

每组 shape 仅有 6 个重复测量值，样本量不足以进行严格的 t-test 或 Wilcoxon 符号秩检验。0.16% 的全局均值差异视为**提示性而非结论性**。结论主要基于：

- 跨 Shape 的一致趋势（81.2% 的 Run-Group 组合三流占优）
- 大差异项的可复现性（Group 3 并行劣势 6/6 一致，Group 7 顺序优势 6/6 一致）

### 4.5 Benchmark 方法说明及改进建议

当前方法：每个数据点执行 500 次 Graph 模式迭代取平均值，总测量时间约 0.5s。

此时间窗口较短，GPU 频率波动和背景噪声可能造成 1~2% 的方差。建议下一版测试：

- 迭代次数从 500 提升到 **5000**
- 丢弃前 1000 次（消除冷启动）
- 取后 4000 次的**中位数**作为代表值

## 五、工程考量

### 5.1 三流并行的硬性约束：梯度不能覆盖输入

三流并行的拓扑中，DGrad 和 WGrad 并发执行：

```
S2(DGrad): d_grad_conv_out, d_weight → 写入 d_grad_input
S3(WGrad): d_grad_conv_out, d_input    → 读取 d_input
```

如果 `d_grad_input` 和 `d_input` 共用同一块内存（即梯度覆盖输入，这是现有 Compiler 的设计），DGrad 写入 `d_grad_input` 的同时 WGrad 正在读取 `d_input` → **数据竞争，结果不可预测**。

要支持三流并行，必须：

- 每个 Conv 层的输入特征图显存翻倍（`d_input` + 独立的 `d_grad_input`）
- 修改 Compiler 的整个反向传播内存分配逻辑
- 可能引入更多内存碎片和分配开销

### 5.2 三流顺序完美兼容梯度覆盖

三流顺序的拓扑天然保证 WGrad 在 DGrad 之前完成：

```
S3(WGrad): d_grad_conv_out, d_input → 读取 d_input → 写入 d_grad_weight
                   ↓ event 同步
S2(DGrad): d_grad_conv_out, d_weight → 写入 d_grad_input
                                           ↑
                    WGrad 已完成对 d_input 的读取，d_grad_input 可以安全覆盖 d_input
```

**不需要任何 Compiler 改动，零工程代价。**

### 5.3 决策矩阵

| 维度 | 三流并行 | 三流顺序 | 单流 |
|------|---------|---------|------|
| 性能（全局均值） | 1.4124 ms | **1.3989 ms** | 1.4011 ms |
| 性能（全局中位数） | 1.1775 ms | **1.1480 ms** | 1.1530 ms |
| 兼容梯度覆盖 | ✗ 需独立显存 | ✓ 天然兼容 | ✓ 天然兼容 |
| Compiler 改动 | 大（内存模型重构） | **零** | 零 |
| 显存开销 | +1× 输入特征图 | 无额外开销 | 无额外开销 |
| 代码复杂度 | 高（fork/join） | 中（顺序依赖） | 低 |

## 六、结论

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│  推荐方案：三流顺序 (cbr_bwd_fp16_seq)                            │
│                                                                  │
│  1. 全局性能最优（均值 1.3989ms，中位数 1.1480ms）                 │
│  2. 三流并行与之性能等价（同策略噪声级差异）                        │
│  3. 完美兼容现有 Compiler 的梯度覆盖设计                           │
│  4. 零工程改动代价                                                 │
│  5. 相比单流，81% 的配置组合有性能优势                             │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

三流并行的额外工程代价（独立显存、Compiler 重构）无法从性能上获得回报——因为它并不比三流顺序更快。三流顺序是性能、工程复杂度、兼容性的最优平衡点。
