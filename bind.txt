以下内容与原方案有所不同，如有冲突，请以以下内容为准。
我们的Preprocessor的配置，是有一个严格顺序的，用一个状态机来控制。在PreprocessorConfigured（完成config_preprocessor）之后，是TransformsSet（完成set_train_transforms和set_val_transforms）。
现在，我们要在这两个状态之间，插入一个新的状态，叫DeviceConfigured，它对应的方法是config_device。config_device有三个重载方法，只要其中一个完成，就算进入DeviceConfigured状态。
这三个重载方法是：
1、config_device(const std::string& engine_device, bool auto_cpu_binding = true)
2、config_device(const std::string& engine_device, const std::vector<int>& gpu_ids, bool auto_cpu_binding = true)
3、config_device(const std::string& engine_device, const std::string& gpu_id_str, bool auto_cpu_binding = true)
engine_device支持“CPU”、“GPU”、“CUDA”、“MUSA”这几种，方法内部会自动转化为全大写。如果用户输入的是“CUDA”或“MUSA”，会自动转化为“GPU”。
如果用户选择的是CPU，那么，相当于选用的GPU列表为空，CPU核绑定也可以直接跳过。
一旦用户选择的是GPU，就要立刻探测可见的GPU总数。
如果用户采用第1种方法输入，那就意味着选用所有可见的GPU（当然，如果没有可见的GPU，那就是选用的GPU列表为空，自动转化为选用CPU，即使用户要求使用GPU，但这样的话需要给个warning）。
如果用户采用第3种方法输入，比如config_device("GPU", "0,1,2,7")，那就要解析字符串里的数字，根据逗号分隔，去掉所有空格，获取到里面的数字后，去掉重复的数字，然后放进vector里面，如果出现了超出可见GPU总数范围的数字，那就要抛出异常ValueError。
如果用户采用第2种方法输入，那就要立刻检测合法性，去掉重复的数字，如果出现了超出可见GPU总数范围的数字，那就要抛出异常ValueError。
此外，我们要求用户选定的GPU数是2的幂，如果不是，也要抛出异常。
当用户确定了CPU/GPU，并且指定了GPU编号，就看auto_cpu_binding。如果当前的平台不是TR_SCENCE_GPU_CLOUD，那这个参数就没用，不管是true还是false都不执行绑核。
如果当前平台是TR_SCENCE_GPU_CLOUD且auto_cpu_binding为true，那就要确定绑核策略。
确定绑核策略的方法必须跟tests/bind/a.cpp完全一致，并且要用std::cout打印哪个PW（或线程ID）绑定到了哪个CPU核。
确定完绑核策略后，向GlobalRegistry注册有关变量，这个方法就算结束了，绑核并不在这个方法内进行。
但是这个方法可以为Preprocessor确定以下几个成员变量（并向GlobalRegistry注册）：模型训练推理是使用CPU还是GPU、实际使用的GPU数（必须与前面设定的world_size一致）、实际使用的GPU编号、是否绑核（只要auto_cpu_binding为false或TR_SCENCE_GPU_CLOUD没有定义，就是不绑核）、每个线程所要绑定的CPU核的ID（这个应该用数组或vector存储，要支持多线程无锁访问不冲突）。
再次强调，绑核的结果必须与tests/bind/a.cpp严格一致。

这里需要注意的是，跟原方案不同的地方在于，我们不再设立AUTO_CPU_BINDING宏，而是使用bool auto_cpu_binding = true标志位。
绑核操作应该是在预处理多线程首次展开后、PW的工作区内存分配之前。我们必须确保线程join并重启后，相同的线程ID依然对应相同的CPU核、相同的PW。
再次说明，当且仅当auto_cpu_binding为true且TR_SCENCE_GPU_CLOUD已定义的时候，会执行这个绑核操作。否则就跳过。
每个线程执行绑核的时候，也要打印一行字，简要说明XX线程绑定到了XX CPU核。
