# 【今日话题：银行家舍入】

## 【题目】

我们的预处理的最后一步，是进行像素的归一化并进行C通道填充，最后根据AMP的要求转为FP16。

毫无疑问我们会在归一化中使用FP32来进行运算，最后再根据需要转为FP16。

但问题是，最后的FP32→FP16的转换，应该使用截断舍入还是银行家舍入（RNE）？RNE真的必要吗？

我们有两个版本的测试样例tests/model/normalize.cpp和tests/model/normalize_rne.cpp，两者的差别仅在于FP32→FP16的实现。





## 【小伙伴S】

● 现在让我分析这个关键问题：作为ResNet-50的AMP训练输入，是否有必要使用RNE？

  我的分析观点

  1. 两个normalize程序的实际差异

  从代码分析来看：
  - normalize.cpp: 使用简化的截断策略（去掉RNE舍入逻辑）
  - normalize_rne.cpp: 使用完整的银行家舍入（RNE）策略

  但关键发现是：两者的硬件实现完全相同！
  #if HAS_F16C
      __m128  v32 = _mm_set_ss(f);
      __m128i v16 = _mm_cvtps_ph(v32, _MM_FROUND_TO_NEAREST_INT | _MM_FROUND_NO_EXC);
      return static_cast<std::uint16_t>(_mm_cvtsi128_si32(v16));
  #endif

  都使用了_MM_FROUND_TO_NEAREST_INT，这正是RNE舍入模式！

  2. 在实际训练中的作用分析

  对于ResNet-50的AMP训练输入：

  不需要专门的RNE版本，原因如下：

  硬件层面

  - 在支持AVX2/F16C的现代CPU上，两个程序都使用相同的硬件指令_mm_cvtps_ph
  - 该指令本身就已经实现了RNE舍入
  - 软件回退版本只在很老的硬件上才会用到

  训练动态范围

  - 深度学习训练是迭代的、容错的
  - FP16的精度损失（0.001-0.01）相对于训练过程中的梯度更新噪声来说可以忽略
  - 梯度下降算法会自动适应和补偿这些微小的精度差异

  统计特性

  - 训练数据是随机的、大量的
  - RNE的统计无偏性在大规模数据集上表现并不明显优于简单舍入
  - 对于batch normalization、dropout等正则化技术，舍入方式的影响微乎其微

  3. 实际建议

  对于ResNet-50的AMP训练输入：

  1. 使用normalize.cpp就足够了

    - 在现代硬件上使用RNE硬件指令
    - 代码更简单，维护成本更低
    - 精度完全满足训练需求
  2. normalize_rne.cpp的价值

    - 主要用于老硬件平台或需要严格RNE的特定场景
    - 作为算法正确性的参考实现
    - 不需要集成到训练流程中
  3. 关键点

    - 两个程序在硬件层面没有差异
    - 软件回退的差异在实际训练中影响极小
    - 训练的容错性远大于舍入方式的差异

  结论

  没有必要在AMP训练中专门使用RNE版本。normalize.cpp已经提供了足够的精度，而且在现代硬件上实际上使用的是相同的RNE指令。no
  rmalize_rne.cpp更像是算法正确性的参考实现，适合用于验证和老旧硬件兼容性，但对于实际的ResNet-50
  AMP训练来说，normalize.cpp完全够用。

## 【小伙伴D】

### 分析：ResNet-50 AMP 训练需要 RNE 吗？
### 一、先定性：截断 vs RNE 在数值上的差异
normalize.cpp 的截断方案在规格化正常数范围（占归一化后值的绝对主体）中直接丢弃低13位：

```
截断：    always round down（始终向下取整）  → 系统性负偏差
RNE：     round to nearest, ties to even    → 无偏
```
截断方案引入的偏差是 有方向性的 ——每个被转换的数值都只会被舍入到更小或相等的方向，不会反过来。这种单向偏差不会被后续运算的随机噪声"平均掉"。

### 二、但这偏差有多大？
normalize.cpp 自己的注释（第98-101行）写了："平均相对误差仅约 0.03%"。换算成绝对值：归一化后的 ImageNet 数据典型落在 [-3, 3]，截断造成的绝对偏差约为 0.001 量级。

作为对比，ResNet-50 训练中的其他噪声源：

- SGD 小批量采样的梯度噪声：远大于此
- 数据增强的随机性（flip, crop, 我们自己的 random erasing）：输入级噪声就远超 0.03%
- FP16 前向计算本身的舍入误差：每层都在发生
训练固有的随机性把这些量级的偏差完全淹没了。

### 三、训练 vs 推理的关键区别
这是很多人混淆的地方。AMP 训练与 AMP 推理不同：

AMP 推理 AMP 训练 权重存储 FP16 FP32 master copy 前向精度 FP16 FP16 梯度累积 不涉及 FP16 → FP32 权重更新 不发生 FP32 精度 偏差累积 前向误差即最终误差 权重 master 在 FP32 中，截断偏差不累积进权重

训练维护了 FP32 的 master weight，输入数据上的 0.03% 截断偏差在反向传播中转化为微小的梯度扰动，而权重更新在 FP32 中完成—— 截断偏差不会跨迭代累积到权重空间中 。

### 四、硬件现实
在有 F16C 的硬件上（绝大多数现代 x86 CPU），两条路径 完全相同 ——都走 _mm_cvtps_ph ，其本身就是 RNE。软件回退路径只在无 F16C 的 CPU 上才会走到。而在那种环境下，通常根本不会跑 ResNet-50 训练。

### 五、结论
维度 评价 训练收敛性 截断和 RNE 没有可测量差异 数值纯粹性 RNE 更优（无系统性偏差） 实际场景 两条路径在几乎所有训练机器上 硬件路径完全相同 工程建议 既然 RNE 实现已经没问题，多用几行位运算的代价可忽略， 建议用 RNE ——更多的是"为什么不呢"而不是"必须"

一句话总结：对 ResNet-50 AMP 训练，RNE 不必要，但它是更端正的做法。 截断方案实际运行中不会损害训练收敛性（FP32 master weight + 训练固有噪声起主导作用），而 RNE 消除了唯一可避免的系统性偏差，代价是零（硬件路径本就用 RNE）到极小（软件回退多几个位运算）。既然我们已经把 RNE 版本写对、验证过、注释完善了，用它是正确的工程选择——但如果你问我"训练能不能用截断版本"，答案是"能，精度不会被影响"。
