# 【二十一、姜总工的新计划】

**（在经历了大约23次修复失败之后，我们无奈地向技术觉醒总工程师姜玉麟汇报了情况。姜总工认真看了我们的代码和报告，给出了以下指示。）**



此前的imagenet的dataloader的开发，非常不顺利，我有责任。

我们相当于已经经历了两轮大的彻底重来。第一轮是因为使用了动态分配，结果在Linux平台频繁撞锁，导致速度明显不如预期。第二轮采用了静态分配策略，速度达到了预期，但是在Linux平台又因为服务器的NUMA架构问题，碰到偶发性的超时。

经过我们团队人员和专家的全力检查，问题已经基本锁定，就是多线程情况下，谁负责ready的问题。**当前问题的根本矛盾在于：NUMA架构的缓存一致性延迟 + 预抢占式调度 + 高频并发同步的三重约束，在现有架构下可能无解。**

我们发现，这个问题只发生在Linux，因为在Windows，MESI协议优化更激进，缓存一致性<100ns，而对于Linux，在NUMA架构下，缓存一致性可能达到数微秒。

重要的是，我们发现，在FULLY模式下，这个问题从未发生过。原因是，FULLY模式只在启动时同步，一旦开始加载，就不再需要同步，最后是使用join的系统级同步，所有线程结束，这样就不会碰到任何竞争；而PARTIAL模式，同步频繁，缓存延迟影响每次同步。

我们已经在这个问题上被困了七天七夜，所有专家提出的意见都尝试过了，包括但不限于删除verify、slot_states原子化、把continue改为sleep，甚至包括最极端的强行等待同步。所有方法，都必定在20次试验之内碰到超时问题。超时问题几乎是我们项目的“死神”。

幸运的是，这个问题经过我们的分析研讨，其实是有最终解法的，但就是需要进行第三次从头设计。

我提出我的分析预判。

首先，静态分配依然是关键思想。这是第一次失败的经验教训。我这里强调，Dataloader的加载线程的责任（哪个线程加载哪个BLOCK，加载到内存的哪个SLOT）和Preprocessor的预处理线程的责任（哪个线程处理哪个样例）都是静态分配的，有且只有一个最简单的动态分配，那就是Preprocessor的预处理线程处理完后的输出放在哪里——这个可以通过一个最简单的fetch_add来实现，即使NUMA架构也不会碰到任何问题。静态分配的做法，可以参考现在的实现，现在的FULLY的实现是比较可靠的、经过检验的。简而言之，就是在读取文件头后，确认了有多少个BLOCK，就确认好哪个线程负责加载哪个BLOCK，而每个线程的槽位是固定的——0号线程是0号槽位，1号线程是16MB偏移量后的1号槽位，以此类推。

其次，就是解决这次PARTIAL模式的超时问题的杀手锏。那就是，我们称为“BATCHED FULLY模式”的新版PARTIAL模式。首先，我们的缓冲区不再使用环形缓冲区。我们回到经典的双缓冲，缓冲区分为A区和B区。我们定义以下变量：第一，加载线程数N。第二，BLOCK大小block_size。第三，预取系数prefetch_factor。对于DTS Loader，BLOCK大小固定为16MB，因为这是文件格式决定的。对于RAW Loader，则必须比16MB大，原因是ImageNet的JPEG原图最大的可能有十几MB，如果BLOCK的大小只有16MB，那就会经常碰到大图片装不下的情况。我们建议，RAW的情况下，BLOCK大小设为64MB。我们推荐，默认的加载线程数是8，这是我们试出来的最优解。但是用户可以选择1、2、4、8、16这几种，如果用户输入的不是2的幂，自动向下取2的幂；如果超过16就自动改为16，不做任何提示。我们依然把N个BLOCK称为一个GROUP。预取系数就是，每一个缓冲区里有多少个GROUP。我们硬性要求，对于线程数N=1的情形，设定的prefetch_factor最小是2，如果用户设定比2小的值，则自动改为2，不做任何报错和提示。对于RAW，prefetch_factor默认是1；而对于DTS，prefetch_factor默认是4。这就意味着，我们每个缓冲区（A区或B区），在DTS模式下都能容纳4个GROUP，大小是4×N×16MB。那么，16线程下，PARTIAL的缓冲区总大小就是2×4×16×16MB=2GB。而对于RAW模式，在16线程下是2×1×16×64MB=2GB，同样是2GB的大小。我们要求单线程时prefetch_factor最小是2，原因是我们要进行样本级随机，如果是1的话，就没法进行BLOCK间的随机了。为什么DTS模式的prefetch_factor默认是4呢？原因是，在我们的新设计中，每次开始加载一个缓冲区，都有重新展开多线程，每次加载完一个缓冲区，都要join，这样是会有比以前的环形缓冲区更大的开销，太小的prefetch_factor的话，这个开销的影响会很大。事实上我们也会考虑内存允许的情况下要使用尽可能大的prefetch_factor。不过，prefetch_factor不要求2的幂，它只要是正整数就行。程序开始，我们的N个加载线程先加载A区。注意，在prefetch_factor＞1的情形下，每个线程的槽位依然是相连的，比如说，DTS模式下的prefetch_factor是4，那么0号线程的4个槽位是连在一起的，总共4×16MB=64MB。在这64MB之后，才是1号线程的槽位。由于我们采用静态分配的策略，每个线程从一开始就知道自己应该加载哪个BLOCK、应该加载到哪个槽位，完全不存在竞争。注意，现在我们不再使用skip_first（第一个epoch不随机）了，只要用户选择shuffle，那就是每个epoch都随机。这里的打乱策略，我们可以沿用以前的。我们不再采用“每个GROUP进行样本级打乱”的策略，而是在加载完了整个A区（或B区）之后，再把整个A区（或B区）进行样本级打乱。刚开始，缓冲区状态是EMPTY，然后，所有GROUP加载完成，缓冲区状态是LOADED，然后，整个缓冲区进行样本级打乱，缓冲区状态变为READY，在预处理器把整个缓冲区都读取完毕后，缓冲区状态变回EMPTY，所有内存变为可以复用。如你所见，A区或B区各有一个buffer_state，这个状态其实是在多线程join之后才改变的。这样就完全不存在一个查询时的竞争、同步责任丢失的问题，因为状态的改变和查询都是在多线程结束后。在数据加载器加载完A区并打乱后，A区设为READY，然后数据加载器就去加载B区，因为此时B区是EMPTY，同时，预处理器就去A区读取数据。接下来就是一个等待的过程，必须在预处理器读取完A区、且数据加载器加载完B区之后，两者进行SWAP。先完成的那个就先等着。这里不会引入与之前一样的同步问题，因为多线程结束了。

对于RAW，我们同样引入了BLOCK的概念，只不过这个BLOCK跟DTS的固定16MB的BLOCK不一样，它的含义是每个线程所负责的内存，对应的也就是SLOT。我们的思路是，每个线程往它的SLOT里面一张张地加载图片，直到图片的size放不下为止，那就下一次再加载。

接下来讲FULLY模式。此前我们的FULLY模式是成功的，按理说不需要做多少修改。但是也还是要修改一点点，因为我们的乱序策略变了——变为prefetch_factor个GROUP之后乱序，而不是每个GROUP乱序。乱序完成后，那几个GROUP才可以被读取。所以，事实上，FULLY模式下的state也变成了prefetch_factor个GROUP共用一个state。

这里说一下RAW的FULLY模式，它申请的内存应该参考DTS的FULLY模式，LV0就是原图的大小，所以DTS需要多大，RAW就需要多大。我的意思是，对于RAW的FULLY模式，所需要预分配的内存大小就根据经验硬编码下来。没记错的话，训练集至少要140288MB，验证集至少要6416MB。注意尾部不完整的BLOCK的情况。

最后我们讲预处理器preprocessor。它的线程数是M。这里我们尝试取消数量的限制和“2的幂”的限制，但要求M是GPU数量U的整数倍。我们对预处理器的责任进一步进行限定，那就是必须完全按顺序领取样本。不管是PARTIAL模式还是FULLY模式，我们的样本在打乱之后都有个顺序。M个预处理线程中，假如M是8，那么，0号线程必定领取0号样本，1号线程必定领取1号样本，7号线程必定领取7号样本。在那之后，0号线程领取8号样本，1号线程领取9号样本，以此类推。如果5号线程领取完了缓冲区的最后一个样本，那么，在下一个缓冲区变为READY时，第一个样本应该由6号线程领取。一定一定要遵循这种严格的“排队领取”原则，因为我们最后会按线程ID来划分责任，每个线程会把它领取和处理的样本放在固定的位置，如果没有这样的顺序的话，就会出现各线程领取的样本数不均的问题。按照我们现在的思路，不管是PARTIAL还是FULLY模式，最后每个预处理线程领取到的样本总数几乎是一样的，最多相差1。

注意我们这里的类的名称已经是preprocessor了，而不是preprocessoremulator。我的意思是，彻底抛弃原来的preprocessoremulator，写全新的类，因为规则完全变了。现在这个preprocessor的功能依然是，每个线程把它读取到的样本的size和标签记录下来，写成“线程ID,样本有效字节数,标签”的格式，保存为csv，文件以“worker_<线程ID>”命名。

我们依然要看到随机可复现性，这在今天已经不再是个问题，因为我们已经知道了可以用静态分配的方法把一切都提前确定好。随机数的发生依然取决于philox.h和rng.h，这个不再赘述。理论上如果全局的种子修改，整个读取的结果也会修改；全局的种子固定，那么所有的顺序也必定固定，跟读取的速度完全无关，我们希望看到的是，多次运行之下，preprocessor输出的csv文件依然是完完全全一样的。

我们实现的顺序，依然跟以前的计划一样，先实现ImageNet的DTS Loader，然后验证读取的样本总数，验证读取速度（这同时要验证不发生超时错误），然后用preprocessor验证“三大可复现性”：一是不shuffle的情况下的可复现性（FULLY/PARTIAL），二是shuffle的情况下的可复现性（FULLY/PARTIAL），三是FULLY模式下不shuffle的时候的epoch间可复现性（每个epoch从内存中读取到同样的顺序）。

总结起来，我们的最主要改版就是：PARTIAL模式环形缓冲区改为双缓冲、状态改为每个缓冲区一个、加载完每个缓冲区之后所有线程join、缓冲区大小设定、样本级乱序的单位（从GROUP改为了prefetch_factor个GROUP）、preprocessor的领取样本的行为。这次是真的全静态、无竞争，就连缓冲区的回收复用，也是join之后的，这可以很好地避免各种问题。

以上就是我的思路，一个人难免想得不够全面、到位，也有可能有错漏，有赖各位专家和团队成员补充完善。
