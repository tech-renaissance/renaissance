# 【二十、总工程师姜玉麟的意见】

**（注：本部分内容，如果与前文有冲突，应以本部分内容为准。）**

我们的团队之前实现了一个基本的数据加载器，但是效果不理想，我认为应该重做。

最大的问题，我认为是实现的基本思路偏离了方向，导致读取速度明显不如预期。

我们设计BLOCK的目的，就是想要允许每个线程单独负责一个BLOCK，并行读取，实现任务的解耦。我们设计GROUP的目的，是想要实现一种静态的映射，把某个BLOCK映射到GROUP的固定位置，然后在GROUP内实现洗牌。

现在Windows和Linux的实现效果有差别，最主要还是一个动态的fetch_add的问题，在Linux下有更严重的竞争，导致时间大大加长，速度远远不及基础的methed2_native.cpp。

我和专家SN讨论过，静态的顺序分配就是避免锁竞争的最好办法——**每个线程从一开始就已经知道了它要加载哪个BLOCK到哪个位置，怎么还会竞争？**

之前的类设计，太过混乱，我在这里给出一个规范建议：

1、抽象类DataLoader：所有数据集的数据加载器的基类。

2、ImageNetLoader、MnistLoader、CifarLoader：DataLoader类的子类，分别负责不同数据集的加载。其中CifarLoader可以加载CIFAR-10和CIFAR-100，因为这两个数据集基本只有标签的数字不同。注意，这几个类依然是抽象类。

3、ImageNetLoaderRaw、ImageNetLoaderDts：Raw版读取的是ImageNet文件夹的原版图片，是散装的JPEG；而Dts版读取的是我们的.dts压缩文件，包括LV0~LV3。

4、MnistLoaderRaw、MnistLoaderDts：同理，Raw版读取的是下载来的原版MNIST数据集，允许下载、解压，每次都从ubyte文件读取；而Dts版读取的是.dts文件，但这个.dts文件与ImageNet有所不同，它不再是JPEG的拼接，而是int8的像素字节流。

5、CifarLoaderRaw、CifarLoaderDts：分别读取原版的bin文件和.dts。.dts文件也是int8的像素字节流。

这里我们对DataLoader类的功能和接口做一个基本定义。它和它的子类都必须是线程安全的全局单例类。它必须可以返回数据集的名称（“ImageNet”、“MNIST”、“CIFAR-10”、“CIFAR-100”）。它负责管理加载的线程（dataload workers），然后，对后续的Preprocessor暴露多线程读取接口和可读取标志。你可以对它设置加载的线程数和预处理的线程数。然后，你可以分别设置训练集和验证集（或测试集）的路径。我们这里统一叫验证集。另外，你可以设置乱序与否（shuffle），训练集默认乱序，验证集默认不乱序。

对于ImageNetLoader，你可以设置skip_first：意思是第一个epoch不乱序。

我们必须强调，对于MnistLoader和CifarLoader，都是**直接全量加载**，不需要用户选择。但对于ImageNetLoader，你可以选择“FULLY”或“PARTIAL”——前者是全量加载，把整个数据集一次性加载到内存中，在整个训练过程中一直保留；后者是部分加载，使用环形缓冲，每次只加载一部分，然后用完就覆盖新的数据，把内存空间循环利用。我们的最终演示，其实用的必须是FULLY模式，因为我们的服务器的云盘太慢了，每个epoch反复加载必定毁掉整个训练的性能。这里FULLY模式就需要考虑到所使用的数据集的版本，是Raw，还是LV0，还是LV1~LV3？不同版本所占的内存是不一样的。特别强调一句，我们允许三种组合：训练集和验证集都FULLY（各占一块内存，都使用完整缓冲）；训练集PARTIAL、验证集FULLY（训练集使用环形缓冲，验证集使用完整缓冲）；训练集和验证集都PARTIAL（注意！这时是**两者共用一个环形缓冲**！）。如果用户使用了“训练集FULLY、验证集PARTIAL”的违规设定，不要报错，默认转换为“训练集和验证集都是FULLY”的设定。

我们这里说一下，我们跳过CRC-32验证的实现，因为这真的真的不重要。

接下来是两大重点：一个是**多线程**，一个是**随机可复现性**。

多线程关键是避免锁竞争，方法就是静态分配，让每个线程从一开始就知道自己应该读取硬盘中的哪一个BLOCK，并且把它读取到内存的哪个位置（读取到内存的哪个位置是取决于线程ID的，而读取硬盘中的哪个BLOCK取决于它被分配的列表）。这里更新一下GROUP的概念：假如有N个线程（dataload workers），那么内存里N个BLOCK就组成一个GROUP，它们在内存上连续。然后，我们的环形缓冲区就是8个GROUP的大小（不管N=1还是N>1）。我们允许的N是1、2、4、8、16，那么环形缓冲区最大也就是8×16×16MB=2GB。关于随机，我们的做法是三重随机——一是在制作DTS文件时就已经把样本随机，二是在加载之前会为每个线程静态分配读取列表，这个列表是随机的；三是样本级随机。样本级与前面的版本有一个关键的区别！那就是现在统一为“每加载完2个GROUP就执行一次样本级随机”。比如，我们的GROUP有0~7号，那么我们在0号和1号GROUP完全加载完后，对这两个GROUP内的所有样本进行一次随机乱序；然后在2号和3号GROUP都完全加载完后，再对这两个GROUP内的所有样本进行一次随机乱序。最后对于不完整的GROUP，比如末尾只剩0.7个GROUP或1.3个GROUP，那就对剩余的样本进行一次随机乱序。

对于Raw模式，因为没有BLOCK的概念，情况有所不同。它首先是扫描文件夹，记录每个文件夹里的文件名。对文件夹名称进行排序，训练集和验证集是相同的顺序，按0~999标号。然后，乱序的时候，是一次性的、全局随机乱序。而且，Raw模式显然不需要进行Dts那样的“每两个GROUP一次样本级随机”，因为它从一开始就已经完成了样本级的随机。

关于向Preprocessor提供接口，我们也有明确的规定。那就是，我们提供4个东西：标签（int32_t）、图片起始位置指针、图片字节数、可读取标志（ready）。所谓ready，就是所在的GROUP加载完成并且已经执行完了样本级乱序。这里强调一下，Preprocessor的线程数跟DataLoader加载的线程数是可以不一样的（而且通常不一样），DataLoader的加载线程数是N，而Preprocessor的预处理线程数是M。我们的DataLoader必须提供M个“窗口”，每个窗口对应一个Preprocess Worker。这个worker每次都从同一个窗口来取它的数据。如果窗口没有准备好，它就等着。我们分配给窗口的数据也不是“哪个样本先准备好就先放上去”，而是严格按照样本级乱序后的顺序。这是为了确保随机可复现性。是的，对于Raw模式，因为没有GROUP的洗牌，所以一个样本加载完成即为ready。

关于Raw模式的环形缓冲区，我认为大小的设定依然可以参考Dts模式，同样是取决于dataload workers数（N），大小是8×N×16MB。毫无疑问，当读取的新的图片的大小放不下的时候，就回绕到环形缓冲区的开头。我的意思是，虽然Raw模式并不是按BLOCK/GROUP加载，但它所申请的内存依然可以以BLOCK/GROUP为单位。

对于Dts模式和Raw模式，假如环形缓冲区的一个GROUP内仍有正在被Preprocessor读取的指针，那就不能覆盖这个GROUP，必须等待，直到整个GROUP可写。

最后是关于随机可复现性的验证，我们是很重视的。验证方法就是，我们设计一个多线程的PreprocessorEmulator，它通过日志的方式，记录每个preprocess worker读取到的图片的字节数。每一行先输出preprocess worker的ID，然后逗号，然后是它所读取的那张图片的字节数，然后换行。我们期望看到的效果是，读取完训练集后，数据总共有1281167行（等于训练集样本总数），然后如果我们对ID进行排序，看到的图片字节数也总是相同的顺序。这就是随机可复现性了。每个worker打印日志的时间也就模拟了图片预处理的时间。这样可以很好地验证我们想要的功能。

总结起来，我最强调的思路就是——GROUP机制、静态分配机制。关键理念就是减少锁竞争，以及在设计上每一步都确保随机可复现性。

实现的做法可以有很多，请其他专家补充，不一定要拘泥于我前面所说的做法。但是**并行读取的速度**和**随机可复现性**是我们的硬指标。速度我们会与method2_native.cpp对齐（我们做过很多实验，那是并行读取的最高速度），而随机可复现性我们就是用PreprocessorEmulator打印的日志来判断。

最后更新一下数据，那就是我们在最终演示的机器上已经做过测试（method2_native.cpp），最高读取速度是16线程实现的，平均是**2.7MB/s**。如果是8线程那就是**2.6MB/s**。毫无疑问我们没有必要支持32线程以上，因为挤占了预处理资源，而且加载的速度提升并不明显。

实现的路径，应该是：抽象类DataLoader→ImageNetLoader→ImageNetLoaderDts→PreprocessorEmulator→ImageNetLoaderRaw→其他。

测试样例一定一定要验证读取速度和读取得到的样本总数。随机可复现性则是输出日志，后面我检查日志来核对验收即可。另外，我依然认为Windows和Linux应该是相同的读取逻辑，只不过底层实现上可以像method2_native.cpp那样有所区别。
